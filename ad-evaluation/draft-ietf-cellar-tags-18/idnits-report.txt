85	1.  Introduction

87	   Matroska is a multimedia container format defined in [RFC9559].  It
88	   can store timestamped multimedia data but also chapters and tags.
89	   The Tag elements add important metadata to identify and classify the
90	   information found in a Matroska Segment.  It can tag a whole Segment,
91	   separate Tracks elements, individual Chapter elements or Attachments
92	   elements.

94	   Some details about tagging are already present in Section 24 of
95	   [RFC9559].  Readers of this document should be familiar with that
96	   section, the different high level parts of Matroska as defined in in
97	   Section 4.5 of [RFC9559] and EBML Master Elements as defined in in
98	   Section 7.7 of [RFC8794].

100	   While the Matroska tagging framework allows anyone to create their
101	   own custom tags, it is important to have a common set of values for
102	   interoperability.  This document defines a set of common tag names
103	   used in Matroska.

105	2.  Notation and Conventions

107	   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
108	   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
109	   "OPTIONAL" in this document are to be interpreted as described in BCP
110	   14 [RFC2119] [RFC8174] when, and only when, they appear in all
111	   capitals, as shown here.

113	3.  Tagging

115	   When a SimpleTag is nested within another SimpleTag, the nested
116	   SimpleTag becomes an attribute of its parent SimpleTag.  For
117	   instance, if you wanted to store the dates that a singer started
118	   being the lead performer, then your SimpleTag tree would look
119	   something like this:

121	   *  Targets

123	      -  TagTrackUID = {track UID of tagged content}.

125	   *  ARTIST = "Pet Shop Boys"

127	      -  LEAD_PERFORMER = "Neil Tennant"

129	         o  DATE_STARTED = "1981-08"

131	   This corresponds to this layout of EBML elements:

133	   <Tags>
134	     <Tag>
135	       <Targets>
136	         <TagTrackUID>{track UID of tagged content}</TagTrackUID>
137	       </Targets>

139	       <SimpleTag>
140	         <TagName>ARTIST</TagName>
141	         <TagString>Pet Shop Boys</TagString>

143	         <!-- sub tag(s) about the ARTIST -->
144	         <SimpleTag>
145	           <TagName>LEAD_PERFORMER</TagName>
146	           <TagString>Neil Tennant</TagString>

148	           <!-- sub tag(s) about the LEAD_PERFORMER -->
149	           <SimpleTag>
150	             <TagName>DATE_STARTED</TagName>
151	             <TagString>1981-08</TagString>
152	           </SimpleTag>

154	         </SimpleTag>

156	       </SimpleTag>
157	     </Tag>
158	   </Tags>

160	   In this way, it becomes possible to store any SimpleTag as attributes
161	   of another SimpleTag.

163	3.1.  Why Official Tags Matter

165	   There is a debate between people who think all tags should be free
166	   and those who think all tags should be strict.  Our recommendations
167	   are in between.

169	   Advanced-users application might let you put any tag in your file.
170	   But for the rest of the applications, they usually give you a basic
171	   list of tags you can use.  Both have their needs, but it's usually a
172	   bad idea to use custom or exotic tags because you will probably be
173	   the only person to use this information even though everyone else
174	   could benefit from it.  So hopefully, when someone wants to put
175	   information in one's file, they will find an official one that fits
176	   their need and hopefully use it.  If it's not in the list, this
177	   person can try to add a new tag in the Matroska Tags Names registry
178	   (Section 6.1).  This registry is not meant to have every possible
179	   information in a file.  Matroska files are not meant to become a
180	   whole database of people who made costumes for a film.  A website
181	   would be better for that.  It's hard to define what should be in and
182	   what doesn't make sense in a file; thus, each demand needs to balance
183	   if it makes sense to be carried over in a file for storage and/or
184	   sharing or if it doesn't belong there.

186	   We also need an official list simply for developers to be able to
187	   display relevant information in their own design, if they choose to
188	   support a list of meta-information they should know which tag has the
189	   wanted meaning so that other apps could understand the same meaning.

191	3.2.  Tag Formatting

193	3.2.1.  TagName Formatting

195	   Official TagName values MUST consist of UTF-8 capital letters,
196	   numbers and the underscore character '_'.

198	   Official TagName values MUST NOT contain any space.

200	   Official TagName values MUST NOT start with the underscore character
201	   '_'; see Section 3.1.

203	   It is RECOMMENDED to start a tag name with the underscore character
204	   '_' for non official tags than are not meant to make it to the list
205	   of official tags.

207	3.2.2.  TagString Formatting

209	   Although tags are metadata mostly used for reading, there are cases
210	   where the string value could be used for sorting, categorization,
211	   etc.  For this reason, when possible, strict formatting of the value
212	   should be used so everyone can agree on how to use the value.

214	   Multiple items SHOULD never be stored as a list in a single
215	   TagString.  If there is more than one tag value with the same name to
216	   be stored, then more than one SimpleTag SHOULD be used.

218	   Due to preexisting files where these formatting rules were not
219	   explicit, they are usually presented as rules that SHOULD be applied
220	   when possible, rather than MUST be applied at all times.  It is
221	   RECOMMENDED to use strict formatting when writing new tag values.

223	3.2.2.1.  Date Tags Formatting

225	   TagString fields defined in this document with dates MUST have the
226	   following format: "YYYY-MM-DD hh:mm:ss.mss" or a reduced version.
227	   The format is similar to the [ISO8601] date and time format defined
228	   in Appendix A of [RFC3339] without the "T" separator, without the
229	   time offset and with the addition of the milliseconds "mss".  The
230	   date and times represented are in Coordinated Universal Time (UTC).

232	   Date and times are usually not precise to a particular millisecond.
233	   To store less accurate dates, parts of the date string are removed
234	   starting from the right.  For instance, to store only the year, one
235	   would use "2004".  To store a specific day such as May 1st, 2003, one
236	   would use "2003-05-01".

238	3.2.2.2.  Number Tags Formatting

240	   TagString fields that require a floating-point number MUST use the
241	   "." mark instead of the "," mark.  Only ASCII numbers "0" to "9" and
242	   the "." character MUST be used.  The "." separator represents the
243	   boundary between the integer value and the decimal parts.  If the
244	   string doesn't contain the "." separator, the value is an integer
245	   value.  Digit grouping delimiters MUST NOT be used.

247	   To display it differently for another locale, applications MUST
248	   support auto replacement on display.

250	   In legacy media containers, it is possible that the "," character
251	   might have been used as a separator or that digit grouping delimiters
252	   might have been used.  A Matroska Reader SHOULD consider the
253	   following character handling to parse such legacy formats:

255	   *  if multiple instances of the same non-number character are found,
256	      they are be ignored,
257	   *  if only one "." character is found and no other non-number
258	      character is found, the "." is the integer-decimal separator,
259	   *  if only one "," character is found and no other non-number
260	      character is found, the "," is a digit grouping delimiter,
261	   *  any other non-number character is ignored.

263	3.2.2.3.  Country Code Tags Formatting

265	   TagString fields that use a Country Code MUST use the Matroska
266	   countries form defined in Section 13 of [RFC9559], i.e. [RFC5646]
267	   two-letter region subtags, without the UK exception.

269	3.3.  Target Types

271	   The TargetTypeValue element allows tagging of different parts that
272	   are inside or outside a given file.  For example, in an audio file
273	   with one song you could have information about the album it comes
274	   from the CD set even if it's not found in the file.

276	   For applications to know the kind of information (e.g., "TITLE")
277	   relates to a certain level (CD title or track title), we also need a
278	   set of official TargetTypeValue values and TargetType names.  That
279	   also means the same tag name can have different meanings depending on
280	   its TargetTypeValue, otherwise we would end up with 7 "TITLE_" tag
281	   names.

283	   For human readability a TargetType string can be added next to the
284	   corresponding TargetTypeValue.  Audio and video have different
285	   TargetType values.  The following table summarizes the TargetType
286	   values found in Section 5.1.8.1.1.2 of [RFC9559] for audio and video
287	   content:

349	   Tags from a TargetTypeValue apply to the all lower TargetTypeValues.
350	   This means that if a CD has the same artist for all tracks, you just
351	   need to set the "ARTIST" tag at TargetTypeValue 50 (ALBUM) and not to
352	   each TargetTypeValue 30 (TRACK), but you can also repeat the value
353	   for each track.  If some tracks of that CD have no known "ARTIST",
354	   the value MUST be set to nothing, a void string "" as detailed in
355	   Section 24.2 of [RFC9559], so that the album "ARTIST" doesn't apply.

357	   If a tag with a given TagName is found at a TargetTypeValue, only
358	   values of that TagName are valid at that TargetTypeValue level.  In
359	   other words, the TagName values from upper TargetTypeValue levels
360	   don't apply at that level.

362	   Multiple SimpleTag with the same TagName can be used at a given
363	   TargetTypeValue level when each SimpleTag contain a TagString.  For
364	   example this can be useful to find a single "ARTIST" even when they
365	   are found in a collaboration.  The concatenation of each TagString
366	   represents the value for the TagName at this level.  The
367	   presentation, for instance with a separator, is up to the
368	   application.

370	3.3.1.  Target Types Parts

372	   There are three organizational tags defined in Section 4.2:

374	   *  TOTAL_PARTS

376	   *  PART_NUMBER

378	   *  PART_OFFSET

380	   These tags allow specifying the ordering of some tags within another
381	   group of tags.

383	   For example if you have an album with 10 tracks and you want to tag
384	   the second track from it.  You set "TOTAL_PARTS" to "10" at
385	   TargetTypeValue 50 (ALBUM).  It means the "ALBUM" contains 10 lower
386	   parts.  The lower part in question is the first lower TargetTypeValue
387	   that is specified in the file.  So, if it's TargetTypeValue = 30
388	   (TRACK), then that means the album contains 10 tracks.  If
389	   TargetTypeValue is 20 (MOVEMENT), that means the album contains 10
390	   movements, etc.  And since it's the second track within the album,
391	   the "PART_NUMBER" at TargetTypeValue 30 (TRACK) is set to "2".

393	   If the parts are split into multiple logical entities, you can also
394	   use "PART_OFFSET".  For example you are tagging the third track of
395	   the second CD of a double CD album with a total of 10 tracks the
396	   "TOTAL_PARTS" at TargetTypeValue 50 (ALBUM) is "10", the
397	   "PART_NUMBER" at TargetTypeValue 30 (TRACK) is "3", and the the
398	   "PART_OFFSET" at TargetTypeValue 30 (TRACK) is "5", which is the
399	   number of tracks on the first CD.

401	   When a TargetTypeValue level doesn't exist it MUST NOT be specified
402	   in the files, so that the "TOTAL_PARTS" and "PART_NUMBER" elements
403	   match the same levels.

405	   Here is an example of an audio record with 2 tracks in a single file,
406	   corresponding to [DaFunk].  There is one Tag element for the record,
407	   and one Tag element per track on the record.  Each track is
408	   identified by a chapter.

428	      -  TagChapterUID = 12345

430	   *  TITLE = "Da Funk"

432	   *  PART_NUMBER = "1"

434	   The Tag for the second track:

436	   *  Targets

438	      -  TargetTypeValue = 30

440	      -  TagChapterUID = 67890

442	   *  TITLE = "Rollin' & Scratchin'"

444	   *  PART_NUMBER = "2"

597	3.4.  Multiple Targets UID

599	   A Tag element has a single Targets element with a single
600	   TargetTypeValue element.  But the Targets element can contain various
601	   TagTrackUID, TagEditionUID, TagChapterUID and TagAttachmentUID
602	   elements.

604	   When multiple values are found using the same Tag UID element (e.g.,
605	   TagTrackUID) a logical OR is applied on these elements.  In other
606	   words the tags apply to each entity defined by a UID.  This is the
607	   list of UIDs the tags apply to (e.g., list of TagTrackUID).  Such a
608	   list may contain a single UID element.

610	   When different lists of Tag UID elements are found (e.g., a list of
611	   TagTrackUID and a list of TagChapterUID) a logical AND is applied
612	   between those lists.  In other words the tags apply only to the
613	   entities matching a UID in each list of Tag UID elements.

615	   These operations allow factorizing tags that would otherwise need to
616	   be repeated multiple times.

618	   Here is an example of a Tag applying to 2 chapters, using the same
619	   [DaFunk] example as in Section 3.3.1:

621	   *  Targets

623	      -  TargetTypeValue = 30

625	      -  TagChapterUID = 12345

627	      -  TagChapterUID = 67890

629	   *  WRITTEN_BY = "Thomas Bangalter"

631	   *  WRITTEN_BY = "Guy-Manuel de Homem-Christo"

633	   *  PRODUCER = "Thomas Bangalter"

635	   *  PRODUCER = "Guy-Manuel de Homem-Christo"

637	   This corresponds to this layout of EBML elements:

639	   <Tags>
640	     <Tag>
641	       <Targets>
642	         <TargetTypeValue>30</TargetTypeValue>

644	         <!-- chapter with Da Funk -->
645	         <TagChapterUID>12345</TagChapterUID>

647	         <!-- chapter with Rollin' & Scratchin' -->
648	         <TagChapterUID>67890</TagChapterUID>
649	       </Targets>

651	       <!-- first writer of Da Funk and Rollin' & Scratchin' -->
652	       <SimpleTag>
653	         <TagName>WRITTEN_BY</TagName>
654	         <TagString>Thomas Bangalter</TagString>
655	       </SimpleTag>

657	       <!-- second writer of Da Funk and Rollin' & Scratchin' -->
658	       <SimpleTag>
659	         <TagName>WRITTEN_BY</TagName>
660	         <TagString>Guy-Manuel de Homem-Christo</TagString>
661	       </SimpleTag>

663	       <!-- first producer of Da Funk and Rollin' & Scratchin' -->
664	       <SimpleTag>
665	         <TagName>PRODUCER</TagName>
666	         <TagString>Thomas Bangalter</TagString>
667	       </SimpleTag>

669	       <!-- second producer of Da Funk and Rollin' & Scratchin' -->
670	       <SimpleTag>
671	         <TagName>PRODUCER</TagName>
672	         <TagString>Guy-Manuel de Homem-Christo</TagString>
673	       </SimpleTag>
674	     </Tag>
675	   </Tags>

677	   Some combinations of different Tag UID elements are not possible.

679	   A TagChapterUID and TagAttachmentUID can't be mixed because there is
680	   no overlap with a Chapter and an Attachment that would make sense.
681	   An attachment applies to the whole segment and can be tied to tracks,
682	   via \Segment\Tracks\TrackEntry\AttachmentLink as defined in
683	   Section 5.1.4.1.24 of [RFC9559], but not chapters.

685	   Mixing TagEditionUID and TagChapterUID elements is also not useful
686	   because each Chapter UID would need to be in one of the Chapter
687	   Edition UID.  That would be the same as not using the list of
688	   TagEditionUID at all.

690	   The following table shows the allowed combinations between lists of
691	   Tag UID elements:

693	   +============+================+=========+=========+================+
694	   | UID        | Track          | Edition | Chapter | Attachment     |
695	   | elements   |                |         |         |                |
696	   +============+================+=========+=========+================+
697	   | Track      | YES            | YES     | YES     | with matching  |
698	   |            |                |         |         | AttachmentLink |
699	   +------------+----------------+---------+---------+----------------+
700	   | Edition    | YES            | YES     | NO      | YES            |
701	   +------------+----------------+---------+---------+----------------+
702	   | Chapter    | YES            | NO      | YES     | NO             |
703	   +------------+----------------+---------+---------+----------------+
704	   | Attachment | with matching  | YES     | NO      | YES            |
705	   |            | AttachmentLink |         |         |                |
706	   +------------+----------------+---------+---------+----------------+

708	              Table 3: Tag UID elements allowed combinations

718	      -  TargetTypeValue = 30

1621	5.  Security Considerations

1623	   This document inherits security considerations from the EBML
1624	   [RFC8794] and Matroska [RFC9559] documents.

1626	   Tag values can be either TagString or TagBinary blobs.  In both cases
1627	   issues can happen if the parsing of the data fails.

1629	   Most of the time strings are kept as-is and don't pose a security
1630	   issue, apart from invalid UTF-8 values.

1632	   String tags that are parsed like "REPLAYGAIN_GAIN" or
1633	   "REPLAYGAIN_PEAK" defined in Section 4.10 or string tags following
1634	   the rules from Section 3.2.2 or string tags following other strict
1635	   formats like URLs may cause issues when the string is bogus or in an
1636	   unexpected format.

1638	   Binary tags that need to be parsed like "MCDI" defined in
1639	   Section 4.11 may cause issues when the data is bogus or incomplete.

1641	   Due to the nature of nested SimpleTag, it is possible to exhaust the
1642	   memory of the host app by using very deep nesting.  An host app MAY
1643	   add some limits to the amount of nesting possible to avoid such
1644	   issues.

1646	6.  IANA Considerations

1648	6.1.  Matroska Tags Names Registry

1650	   IANA has created a new registry called the "Matroska Tag Names"
1651	   registry.

1653	   To register a new Tag Name in this registry, one needs a Name, a
1654	   Type, a Change Controller, and an optional Reference to a document
1655	   describing the Element ID.

1657	   The Name corresponds to the value stored in the TagName element.  The
1658	   Name SHOULD always be written in all capital letters and contain no
1659	   space as defined in Section 3.2,

1661	   The Type corresponds to which element will be stored the tag value.
1662	   There can be 3 values for the Type:

1664	   *  UTF-8: the value of the Tag is stored in TagString,

1666	   Matroska Tag Names for UTF-8 data are to be allocated according to
1667	   the "First Come First Served" policy [RFC8126].

1669	   *  binary: the value of the Tag is stored in TagBinary,

1671	   Matroska Tag Names for binary data are to be allocated according to
1672	   the "Specification Required" policy [RFC8126].

1674	   *  nested: the tag doesn't contain a value, only nested tags inside.

1676	   Matroska Tag Names for nested tags are to be allocated according to
1677	   the "Specification Required" policy [RFC8126].
