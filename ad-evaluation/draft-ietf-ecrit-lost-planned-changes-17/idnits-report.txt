8	            Validation of Locations Around a Planned Change
9	                draft-ietf-ecrit-lost-planned-changes-17

11	Abstract

13	   This document defines an extension to the Location to Service
14	   Translation (LoST) protocol (RFC5222) that allows a LoST server to
15	   notify a client of planned changes to location data.  This extension
16	   is only useful with the validation function of LoST.  It is
17	   beneficial for LoST validation clients to be aware of planned
18	   changes, since at a known future date, previously valid records may
19	   become invalid, and new records may become valid.  This extension
20	   adds an element to the <findService> request to allow a LoST client
21	   to request validation as of a specified date.  It adds an optional
22	   Time-To-Live element to the response, which informs clients of the
23	   current expected lifetime of a validation.  It also adds a separate
24	   interface to a LoST server that allows a client to poll for planned
25	   changes.  Additionally, this document provides a conventional XML
26	   schema for LoST, as a backwards compatible alternative to the RelaxNG
27	   schema in RFC5222.

78	1.  Introduction

80	   Validation of civic locations involves dealing with data that may
81	   change over time.  A typical example is when a portion of a county or
82	   district that was not part of a municipality is "annexed" to a
83	   municipality.  Prior to the change, a Presence Information Data
84	   Format Location Object (PIDF-LO) [RFC5139] specifying a civic
85	   location in the affected area would have a blank A3 element, or would
86	   contain some other value; after the change, a PIDF-LO specifying the
87	   same location would contain an A3 element set to the name of the
88	   municipality that annexed that part of the county/district.  This
89	   kind of annexation has an effective date and time (typically 00:00 on
90	   the first or last day of a month), known in advance.  Other kinds of
91	   changes may also occur, and these will almost always also have an
92	   effective date that is known in advance.

94	   Records in a LoST client such as a Location Information Server (LIS)
95	   must change around these kinds of events.  Each affected old record
96	   must be discarded, and a new, validated record must be loaded into
97	   the client's database.  It is often difficult for a LIS operator to
98	   know that records must be changed around such events.  There are
99	   other circumstances where locations that were previously valid become
100	   invalid, such as a street renaming or renumbering event.  Using
101	   [RFC5222] validation, the only way for a LoST client such as a LIS to
102	   discover such changes is to periodically revalidate its entire
103	   database.  Of course, this does not facilitate timely changes, is not
104	   coordinated with the actual change event, and also adds significant
105	   load to LoST servers as well as LoST clients such as LISs.  Even when
106	   re-validation is used, a server has no mechanism to control, or
107	   suggest the time period for revalidation.

109	   This extension allows a LoST client to obtain from a LoST server sets
110	   of locations which may change.  It makes use of "partial location
111	   information" which is a set of location elements and values that,
112	   when compared against client location records, identify which of the
113	   client records may change as a result of the planned change.  A set
114	   of such partial locations is termed a "ChangeSet".  ChangeSets have
115	   an ID, and a date when the change is effective.  IDs are ordered so
116	   that changes can be completed in the correct order.  The planned
117	   change interface is a REST/JSON interface that allows a client to
118	   poll a server using the last ID that it obtained from that server.
119	   The response to a poll is a list of all the newer planned changes the
120	   server knows about beyond the ChangeSet whose ID was included in the
121	   poll.  The ID of the last ChangeSet in the returned list is used by
122	   the client in its next poll.

124	   When a LoST client such as a LIS receives a new ChangeSet, it may
125	   prepare one or more new records so that, at the precise planned event
126	   date and time, it may insert the new records into its active database
127	   and delete the old records.  As part of preparing the new records in
128	   advance of the change, a client may use the "asOf" date component of
129	   this extension to perform a LoST validation of the new record as of
130	   the effective date.

132	   The "asOf" date component of this extension allow a LoST client such
133	   as a LIS to be prepared for and smoothly transition to planned
134	   changes.  The polling mechanism allows a client to be informed of
135	   planned changes, while the "asOf" date allows it to verify the
136	   validity of locations before they become active.

138	   Unplanned changes will occur, and periodic revalidation can still be
139	   used to maintain the data in the client's records.  However, LoST
140	   servers should be able to influence the rate of such revalidation.
141	   For this purpose, this extension adds an optional "expires" element
142	   to the <findServiceResponse> to provide advice from a server to a
143	   client as to when revalidation is suggested.  In a
144	   <findServiceResponse>, a LoST server may include the "expires"
145	   element to expressly state when the location should be re-validated.
146	   This avoids clients blindly revalidating every address on an
147	   arbitrary schedule.

149	   There are quite a few implementations of LoST.  Experience with these
150	   implementations indicates that the RelaxNG schema is very difficult
151	   to deal with, both because many commonly used development tools don't
152	   support it, and development staff is often unfamiliar with it.
153	   Informal alternative schemas have been circulated, which is
154	   undesirable as they may not be in conformance with the RelaxNG schema
155	   in [RFC5222].  This document provides an XML schema that replaces the
156	   RelaxNG schema.  It can be used by any implementation interchangeably
157	   with the RelaxNG schema.

170	3.  Planned Change Poll Interface

172	   This document defines a new interface to a LoST server.  The
173	   interface has three endpoints:

175	   1.  Versions, which returns the current version(s) the interface
176	       supports.  This allows the interface to evolve over time.

178	   2.  PlannedChangePoll, which supports a polling mechanism.  The poll
179	       returns a list of changeSetIds which identify ChangeSet objects.

181	   3.  GetChangeSet, which accepts a changeSetId and returns the
182	       identified ChangeSet object.

184	   A ChangeSet object contains an identifier (changeSetId), a date
185	   (changeSetEffective) and an array of partial locations.

187	   A partial location is an array of location information element
188	   namespace/name and value pairs.  A LoST client (such as a LIS)
189	   compares the location elements with its records.  For each of the
190	   client's records where all of the location elements provided in the
191	   partial location have the same values as those in the partial
192	   location, that client record may be affected by the planned change.
193	   The client's records may have other location elements, but those are
194	   not considered in the comparison.  So, for example, a partial
195	   location may have Country, A1, A2, A3 and A4 location elements, which
196	   means that any address with those Country, A1, A2, A3 and A4 values
197	   may be affected by the planned change, regardless of street name and
198	   number.  As another example, a partial location with Country, A1, A2,
199	   A3, A4, RD and POD but not HN applies to any address number on the
200	   specified street.

202	   Servers supporting this extension maintain an ordered list of
203	   changeSetIds.  A changeSetId is a string, which might not show the
204	   ordering.  For example, the ID could be a hashed timestamp, or a
205	   hashed sequence number, among other things.  Given a changeSetId
206	   returned in a prior poll, a server can identify which ChangeSets it
207	   has that come next, in order, after the specified changeSetId.  The
208	   effective date of a ChangeSet returned by a server need not be in the
209	   future.  Tardy clients might not keep up.  On the other hand, a
210	   server is not obligated to keep ChangeSets whose changeSetEffective
211	   date has passed by more than some arbitrary time.  A 12 month time
212	   period may be appropriate for a server whose service area doesn't
213	   have many changes, while a three month time period may be needed in a
214	   service area with frequent changes.  A tardy client in a fast-
215	   changing environment might receive a large number of ChangeSetIds in
216	   response to a poll.

218	   Polls are expected every few minutes.  A new client (or one that has
219	   lost its last changeSetID) performs a poll without specifying an ID,
220	   and the server responds with all the changeSetsIds that it knows
221	   about.  Thereafter, the client retains the last changeSetId from its
222	   most recent poll and uses that in the next poll.  If the response to
223	   that poll contains no changeSetIds, it means the changeSetId the
224	   client has is the latest change the server knows about; the client
225	   uses the same changeSetId in subsequent polls until the server
226	   returns a new changeSetId.

228	   The version mechanism returns a list of versions of the web service
229	   the server supports.  This document specifies version 1.0.  Versions
230	   are described as a major version and a minor version, where major and
231	   minor versions are integers, and a version description is a string
232	   consisting of the major version, a dot, and the minor version.  A
233	   backwards-compatible change within a major version MAY increment only
234	   the minor version number.  A non-backwards-compatible change MUST
235	   increment the major version number.  To achieve backwards
236	   compatibility, implementations MUST ignore any object members they do
237	   not recognize.  Minor version specifications SHALL only add objects,
238	   non-required members of existing objects, and non-mandatory-to-use
239	   functions and SHALL NOT delete any objects, members of objects or
240	   functions.  This means an implementation of a specific major version
241	   and minor version is backwards compatible with all minor versions of
242	   the major version.  The versions mechanism returns an array of
243	   supported versions, one for each major version supported, with the
244	   minor version listed being the highest supported minor version.  When
245	   versions are written or used as a string, the major version is first
246	   and separated from the minor version with a period.  For example
247	   major version 3, minor version 2 is written as "3.2".

249	   The normative definition of the web interface for these endpoints is
250	   contained in Section 8

252	   Discovering the Planned Change Poll interface uses the
253	   "lost+plannedChange" application tag with U-NAPTR using the name
254	   (application unique string) of the LoST server that provided
255	   validation as input along with this new service tag.  Similarly to
256	   LoST, valid protocol tags for use in combination with
257	   "lost+plannedChange" are "http" and "https".  The U-NAPTR responds
258	   with the base URI of the interface and the client must use the value
259	   to replace "localhost" in the yaml in Section 8.

261	4.  <asOf> Element

263	   This document defines a new element in the <findService> request and
264	   <findServiceRespponse> called <asOf>, which contains a date and time
265	   in dateTime format with a required timezone.  A server supporting
266	   this extension validates the location in the request as of the date
267	   and time specified, taking into account planned changes.  This allows
268	   a client to verify in advance that it can make changes in its records
269	   in accordance with changes at the LoST server.

271	   A server responds with what it knows (as of the moment of the
272	   request) will be in effect on or before the <asOf> date

274	   The LoST server is not expected to retain a history of changes.
275	   Therefore, an <asOf> date in the past will return the same results as
276	   omitting it (meaning current data).  There are no constraints on how
277	   far in advance planned changes may occur, and changes to the data may
278	   occur at any time.  Therefore, two queries with the same future
279	   <asOf> value placed at different times can return different results.
280	   The server responds with its current understanding of planned changes
281	   (including any planned or unplanned changes that have already
282	   occurred) and makes no guarantees about future queries of the same
283	   location.

285	   When a server responds with a result <asOf> a time other than the
286	   time of the query, the <findServiceResponse> MUST include <asOf>
287	   containing the time used and each <mapping> included MUST have the
288	   'expires' attribute set to "NO-CACHE".  A server SHOULD NOT include
289	   <asOf> in the result if <asOf> was not requested or if it otherwise
290	   used the current time to formulate the response.  A client receiving
291	   a response containing <asOf> MUST NOT assume that any mappings
292	   returned will remain unchanged and be in effect in the future; a
293	   client wishing to contact a service is still expected to use LoST
294	   contemporaneously with that need.

296	5.  <revalidateAfter> in Response

298	   This extension defines a new element <revalidateAfter> to be added to
299	   the extension point of the <locationValidation> element appearing in
300	   a <findServiceResponse>.  The element contains a date and time after
301	   which the client may wish to revalidate the location with the server.
302	   Alternatively, the element may contain the value "NO-EXPIRATION"
303	   which is an assertion that the server is not currently aware of any
304	   planned changes that would affect the future validation result, but
305	   makes no suggestion as to when the client should revalidate.  A
306	   server MAY add this element to a response when validation is
307	   requested.  A server is not expected to add this element when the
308	   client is asking for validation <asOf> a specified time significantly
309	   into the future.

311	   Selecting a revalidation interval is a complex balancing of
312	   timeliness, server load, stability of the underlying data, and policy
313	   at the LoST server.  Too short, and load on the server may be
314	   undesirably large.  Too long, and invalid data may persist in clients
315	   for unacceptable lengths of time.  The polling mechanism provides
316	   timely notice to synchronize changes, but even with it, it is
317	   advisable for clients to periodically revalidate their records.  In
318	   areas that have little change in data, such as fully built out,
319	   stable communities already part of a municipality, it may be
320	   reasonable to set revalidation periods of six months or longer.  In
321	   areas that are quickly growing, 20-30 day revalidation may be more
322	   appropriate, even though these revalidations might be a majority of
323	   the traffic at the LoST server.
