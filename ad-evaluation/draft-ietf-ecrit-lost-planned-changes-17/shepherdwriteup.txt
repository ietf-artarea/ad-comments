Document Shepherd Write-Up for <a href="/doc/draft-ietf-ecrit-lost-planned-changes/">draft-ietf-ecrit-lost-planned-changes</a>

This template version is dated 4 July 2022.

Document History

(1) Was the document considered in any WG, and if so, why was it not
adopted as a work item there?

This document was submitted by the author to the ECRIT WG, and was
adopted as a work item by that group.

(2) Was there controversy about particular points that caused the WG to
not adopt the document?

The ECRIT WG readily adopted the document as a work item.

(3) Has anyone threatened an appeal or otherwise indicated extreme
discontent? If so, please summarize the areas of conflict in separate
email messages to the responsible Area Director. (It should be in a
separate email because this questionnaire is publicly available.)

There has been no threat of appeal or indication of extreme discontent.
There was robust discussion during development about which of two
possible approaches (push notifications or subscriber polling) was the
more elegant method of implementing the proposed standard, but all
participants agreed that the chosen method was a workable approach.

There was robust discussion on some other relatively minor points,
where differing opinions existed about the better of two choices, but
again all participants agreed that either of the two choices was
workable.  In the end, rough consensus was achieved and the author
proceeded.

(4) For protocol documents, are there existing implementations of the
contents of the document? Have a significant number of potential
implementers indicated plans to implement? Are any existing
implementations reported somewhere, either in the document itself
(as <a href="/doc/rfc7942/">RFC 7942</a> recommends) or elsewhere (where)?

There are no known existing implementations of the contents of this
document.  Several potential implementers are eagerly awaiting this
document's approval.  The NENA i3 standards document makes several
references to a draft version of this document and mandates support
for it.

Additional Reviews

(5) Do the contents of this document closely interact with
technologies in other IETF working groups or external organizations, and
would it therefore benefit from their review? Have those reviews
occurred? If yes, describe which reviews took place.

The contents of this document closely interact with elements of the NENA
i3 standard.  Representatives of that organization have been
fully involved in the ECRIT work leading to this draft document.
There is a great amount of overlap between the membership of the NENA i3
working group and the ECRIT working group.

We do not believe there is any close interaction with technologies in
other IETF working groups.

(6) Describe how the document meets any required formal expert review
criteria, such as the MIB Doctor, YANG Doctor, media type, and URI type
reviews.

No additional formal reviews are required.

(7) If the document contains a YANG module, has the final version of
the module been checked with any of the recommended validation tools
for syntax and formatting validation? If there are any resulting errors
or warnings, what is the justification for not fixing them at this time?
Does the YANG module comply with the Network Management Datastore
Architecture (NMDA) as specified in <a href="/doc/rfc8342/">RFC 8342</a>?

There is no YANG module.

(8) Describe reviews and automated checks performed to validate sections
of the final version of the document written in a formal language, such
as XML code, BNF rules, MIB definitions, CBOR's CDDL, etc.

The XML code contained within the document was reviewed by Jeff Martin,
a member of the ECRIT group who has expertise in XML.  Jeff reported
that the XML code in Sections 6 and 7 of the document "are valid in the
following sense: Both are well-formed XML Schema definitions, and all
the examples in <a href="/doc/rfc5222/">RFC 5222</a> and this new draft are valid per these
sections."

Document Shepherd Checks

(9) Based on the shepherd's review of the document, is it their opinion
that this document is needed, clearly written, complete, correctly
designed, and ready to be handed off to the responsible Area Director?

Yes, there is a definite expressed need for this document.  It is
clearly written, completely addresses the issue it is intended to
address, and is ready to be handed off to the Area Director.

(11) What type of RFC publication is being requested on the IETF
stream (Best Current Practice, Proposed Standard, Internet Standard,
Informational, Experimental or Historic)? Why is this the proper
type of RFC? Do all Datatracker state attributes correctly reflect
this intent?

Proposed Standard is the publication type being requested, as shown in
Datatracker and the header of the latest version of the draft document.
This document extends an existing Proposed Standard (<a href="/doc/rfc5222/">RFC 5222</a>) to
implement a much-needed added function.  The WG was chartered for
standards-track work in this area.
