215	1.  Introduction

217	   Matroska is a multimedia container format defined in [RFC9559].  It
218	   stores interleaved and timestamped audiovisual data using various
219	   codecs.  To interpret the codec data, a mapping between the way the
220	   data is stored in Matroska and how it is understood by such a codec
221	   is necessary.

223	   This document defines this mapping for many commonly used codecs in
224	   Matroska.

226	2.  Notation and Conventions

234	3.  Codec Mappings

236	   A Codec Mapping is a set of attributes to identify, name, and
237	   contextualize the format and characteristics of encoded data that can
238	   be contained within Matroska Clusters.

240	   Each TrackEntry used within Matroska MUST reference a defined Codec
241	   Mapping using the CodecID to identify and describe the format of the
242	   encoded data in its associated Clusters.  This CodecID is a unique
243	   registered identifier that represents the encoding stored within the
244	   Track.  Certain encodings MAY also require some form of codec
245	   initialization to provide its decoder with context and technical
246	   metadata.

248	   The intention behind this list is not to list all existing audio and
249	   video codecs, but rather to list those codecs that are currently
250	   supported in Matroska and therefore need a well defined CodecID so
251	   that all developers supporting Matroska will use the same CodecID.
252	   If you feel we missed support for a very important codec, please tell
253	   us on our development mailing list (cellar at ietf.org).

255	3.1.  Defining Matroska Codec Support

257	   Support for a codec is defined in Matroska with the following values.

259	3.1.1.  Codec ID

261	   Each codec supported for storage in Matroska MUST have a unique
262	   CodecID.  Each CodecID MUST be prefixed with the string from the
263	   following table according to the associated Track Type of the codec.
264	   All characters of a Codec ID Prefix MUST be capital letters (A-Z)
265	   except for the last character of a Codec ID Prefix which MUST be an
266	   underscore ("_").

268	                    +===============+=================+
269	                    | Track Type    | Codec ID Prefix |
270	                    +===============+=================+
271	                    | Video (1)     | "V_"            |
272	                    +---------------+-----------------+
273	                    | Audio (2)     | "A_"            |
274	                    +---------------+-----------------+
275	                    | Complex (3)   | "O_"            |
276	                    +---------------+-----------------+
277	                    | Logo (16)     | "L_"            |
278	                    +---------------+-----------------+
279	                    | Subtitle (17) | "S_"            |
280	                    +---------------+-----------------+
281	                    | Button (18)   | "B_"            |
282	                    +---------------+-----------------+
283	                    | Control (32)  | "C_"            |
284	                    +---------------+-----------------+
285	                    | Metadata (33) | "M_"            |
286	                    +---------------+-----------------+

288	                        Table 1: Codec ID Prefix by
289	                                 Track Type

291	   Each CodecID MUST include a Major Codec ID immediately following the
292	   Codec ID Prefix.  A Major Codec ID MAY be followed by an OPTIONAL
293	   Codec ID Suffix to communicate a refinement of the Major Codec ID.
294	   If a Codec ID Suffix is used, then the CodecID MUST include a forward
295	   slash ("/") as a separator between the Major Codec ID and the Codec
296	   ID Suffix.  The Major Codec ID MUST be composed of only capital
297	   letters (A-Z) and numbers (0-9).  The Codec ID Suffix MUST be
298	   composed of only capital letters (A-Z), numbers (0-9), underscore
299	   ("_"), and forward slash ("/").

301	   The following table provides examples of valid Codec IDs and their
302	   components:

318	3.1.2.  Codec Name

320	   Each encoding supported for storage in Matroska MUST have a Codec
321	   Name.  The Codec Name provides a readable label for the encoding.

323	3.1.3.  Description

325	   An optional description for the encoding.  This value is only
326	   intended for human consumption.

328	3.1.4.  Initialization

330	   Each encoding supported for storage in Matroska MUST have a defined
331	   Initialization.  The Initialization MUST describe the storage of data
332	   necessary to initialize the decoder, which MUST be stored within the
333	   CodecPrivate element.  When the Initialization is updated within a
334	   track, then that updated Initialization data MUST be written into the
335	   CodecState element of the first Cluster to require it.  If the
336	   encoding does not require any form of Initialization, then none MUST
337	   be used to define the Initialization and the CodecPrivate element
338	   SHOULD NOT be written and MUST be ignored.  If the TrackEntry
339	   contains a CodecPrivate element, its data MUST be provided to the
340	   decoder.

342	3.1.5.  Codec BlockAdditions

344	   Additional data that contextualizes or supplements a Block can be
345	   stored within the BlockAdditional element of a BlockMore element
346	   Section 5.1.3.5.2.1 of [RFC9559].  Each BlockAdditional is coupled
347	   with a BlockAddID that identifies the kind of data it contains.

349	   A BlockAddID of 1 means the data in the BlockAdditional element are
350	   tied to the codec.  This BlockAdditional data with a BlockAddID of 1
351	   MAY be passed to the associated decoder alongside the Block content .

353	   A codec definition MUST contain a "Codec BlockAdditions" section if
354	   the codec can use BlockAdditional data with a BlockAddID of 1.

356	   The BlockAddID values are defined in Section 4.

358	3.1.6.  Citation

360	   Documentation of the associated normative and informative references
361	   for the codec is RECOMMENDED.

363	3.1.7.  Superseded By

365	   When a Superseded By is set, the superseding CodecID value MUST be
366	   used instead of the superseded CodecID.

368	   Files MAY exist with the superseded CodecID and MAY be supported by
369	   Matroska Players.

371	3.2.  Recommendations for the Creation of New Codec Mappings

373	   Creators of new Codec Mappings to be used in the context of Matroska:

375	   *  MUST assume that all Codec Mappings they create might become
376	      standardized, public, commonly deployed, or usable across multiple
377	      implementations.

379	   *  MUST employ meaningful values for CodecID and Codec Name that are
380	      not already included in the Matroska Codec IDs Registry, and are
381	      not otherwise known or suspected to be in use, even if they are
382	      not already registered.

384	   *  MUST NOT prefix their CodecID with "X_" or similar constructs.

386	   These recommendations are based on Section 3 of [RFC6648].

388	3.3.  Video Codec Mappings

390	   All codecs described in this section MUST have a TrackType
391	   (Section 5.1.4.1.3 of [RFC9559]) value of "1" for video.  The track
392	   using these codecs MUST contain a Video element -- i.e., EBML Path
393	   \Segment\Tracks\TrackEntry\Video.

395	   Most video codec contain meta information about the data they
396	   contain, like encoded width and height, chroma subsampling, etc.
397	   Whenever possible these information inside the codec SHOULD be
398	   extracted and repeated at the Matroska level with the appropriate
399	   element(s) inside the \Segment\Tracks\TrackEntry\Video and

403	3.3.1.  V_AV1

405	   Codec ID: V_AV1

407	   Codec Name: Alliance for Open Media AV1 Video codec

409	   Description: Only one Sequence Header OBU, as defined in section 6.4
410	   of [AV1], is supported per Matroska Segment.  Each Block contains one
411	   Temporal Unit containing one or more OBUs.  Each OBU stored in the
412	   Block MUST contain its header and its payload.  The OBUs in the Block
413	   follow the Low Overhead Bitstream Format syntax.  They MUST have the
414	   obu_has_size_field set to 1 except for the last OBU in the frame, for
415	   which obu_has_size_field MAY be set to 0, in which case it is assumed
416	   to fill the remainder of the frame.  A SimpleBlock MUST NOT be marked
417	   as a Keyframe if it doesn't contain a Frame OBU.  A SimpleBlock MUST
418	   NOT be marked as a Keyframe if the first Frame OBU doesn't have a
419	   frame_type of KEY_FRAME.  A SimpleBlock MUST NOT be marked as a
420	   Keyframe if it doesn't contains a Sequence Header OBU.  A Block
421	   inside a BlockGroup MUST use ReferenceBlock elements if the first
422	   Frame OBU in the Block has a frame_type other than KEY_FRAME.  A
423	   Block inside a BlockGroup MUST use ReferenceBlock elements if the
424	   Block doesn't contain a Sequence Header OBU.  A Block with
425	   frame_header_obu where the frame_type is INTRA_ONLY_FRAME MUST use a
426	   ReferenceBlock with a value of 0 to reference itself.

428	   Initialization: The CodecPrivate consists of the
429	   AV1CodecConfigurationRecord described in section 2.3 of
430	   [AV1-ISOBMFF].

432	   PixelWidth: MUST be max_frame_width_minus_1+1 of the Sequence Header
433	   OBU.

435	   PixelHeight: MUST be max_frame_height_minus_1+1 of the Sequence
436	   Header OBU.

1795	5.  Audio Codecs

1797	5.1.  WavPack

1799	   WavPack is an audio codec primarily designed for lossless audio, but
1800	   it can also be used as a lossy codec.

1802	   [WAVPACK] stores each data in variable length frames.  That means
1803	   each frame can have a different number of samples.

1805	   Each WavPack block starts with a WavpackHeader header as defined in
1806	   [WAVPACK], stored in little-endian.

1808	   To save space and avoid redundant information in Matroska some data
1809	   from the WavpackHeader header are removed, when saved in Matroska.
1810	   All the data from the WavpackHeader are kept in little-endian.

1812	   The CodecPrivate contains the version 16-bit integer from the
1813	   WavpackHeader of [WAVPACK] stored in little-endian.

1815	   Depending on the number of audio channels and whether the hybrid mode
1816	   is kept or not, the storage of WavPack blocks in Matroska differ.

1818	5.1.1.  Lossless And Lossy Storage

1820	   For multichannel files (more than 2 channels, like for 5.1), a frame
1821	   consists of multiple WavPack blocks.  The first one having the
1822	   INITIAL_BLOCK (bit 11) flag set and the last one the FINAL_BLOCK (bit
1823	   12) flag set.  For a mono or stereo file, both flags are set in each
1824	   WavPack block.

1826	5.1.1.1.  Mono/Stereo

1828	   A Block or SimpleBlock frame contains the following header with the
1829	   some fields taken from the WavpackHeader of a single WavPack block
1830	   followed by the data of that WavPack block.

1832	   {
1833	     uint32_t block_samples; // # samples in this block
1834	     uint32_t flags;         // various flags for id and decoding
1835	     uint32_t crc;           // crc for actual decoded data
1836	   }
1837	   [ block data ]

1839	5.1.1.2.  Multichannel

1841	   For multichannel files, a WavPack file uses multiple WavPack block to
1842	   store all channels of a frame.  The WavPack blocks for each channels
1843	   of a frame are stored consecutively into a Matroska Block or
1844	   SimpleBlock.

1846	   Each WavPack block is preceded by a header.  The header for the first
1847	   WavPack block is similar to the mono/stereo one (Section 5.1.1.1)
1848	   with the addition of a "blocksize" field, which is the size of the
1849	   first WavPack block minus the WavpackHeader size.  The header for the
1850	   following WavPack blocks use the "flags" and "crc" of the
1851	   WavpackHeader of each respective WavPack block, followed with the
1852	   size of each respective WavPack block minus the WavpackHeader size.

1854	   {
1855	     uint32_t block_samples; // # samples in this block
1856	     uint32_t flags;         // various flags for id and decoding
1857	     uint32_t crc;           // crc for actual decoded data
1858	     uint32_t blocksize;     // size of the data to follow
1859	   }
1860	   [ block data # 1 ]
1861	   {
1862	     uint32_t flags;         // various flags for id and decoding
1863	     uint32_t crc;           // crc for actual decoded data
1864	     uint32_t blocksize;     // size of the data to follow
1865	   }
1866	   [ block data # 2 ]
1867	   {
1868	     uint32_t flags;         // various flags for id and decoding
1869	     uint32_t crc;           // crc for actual decoded data
1870	     uint32_t blocksize;     // size of the data to follow
1871	   }
1872	   [ block data # 3 ]
1873	   ...

1875	5.1.2.  Hybrid Storage

1877	   WavPack has a hybrid mode that splits the audio frames between lossy
1878	   and correction packets.  Adding both gives a lossless version of the
1879	   original audio.  It is possible to only store the lossy part in
1880	   Matroska or both together.  Storing only the lossy part is equivalent
1881	   to the format described in Section 5.1.1.  This section explains how
1882	   to store all hybrid data in Matroska.

1884	   Hybrid WavPack is encoded in 2 files.  The first one has a lossy part
1885	   and the second file has the correction part to reconstruct the
1886	   original audio losslessly.

1888	   Each WavPack block in the correction file corresponds to a WavPack
1889	   block in the lossy file with the same number of samples, that's also
1890	   true for a multichannel file.  This means that if a frame is made of
1891	   4 WavPack blocks, the correction file will have 4 WavPack blocks in
1892	   the corresponding frame.  The header of the correction WavPack block
1893	   is exactly the same as in the lossy WavPack block, except for the
1894	   CRC.

1896	   In Matroska, the correction part is stored as an additional data
1897	   available to the Block (see Section 7).  This way a file could be
1898	   remuxed and not keep the Block Additional data and still be usable as
1899	   a lossy WavPack file.  The Block data of the lossy file are stored
1900	   exactly the same as for lossy storage defined in Section 5.1.1.

1902	   A BlockAdditionMapping MUST be used for hybrid WavPack TrackEntry'.

1904	   The BlockAddIDType of that BlockAdditionMapping MUST be set to 1 for
1905	   hybrid WavPack, corresponding to Opaque data; see Section 4.2.2.

1907	   Each WavPack frame is stored in a BlockGroup that MUST have at least
1908	   a BlockMore to hold the correction data.

1910	   The BlockAddID of that BlockMore MUST be 1, i.e., the default value.

1912	5.1.2.1.  Mono/Stereo

1914	   The BlockAdditional element of the correction data BlockMore contains
1915	   the following header with the "crc" field from the WavpackHeader of
1916	   the WavPack block of the correction file matching the WavPack block
1917	   of the lossy frame used to fill the Block data, followed by the data
1918	   of that correction file WavPack block.

1920	   {
1921	     uint32_t crc;           // crc for actual decoded data
1922	   }
1923	   [ correction block data ]

1925	5.1.2.2.  Multichannel

1927	   The BlockAdditional element of the correction data BlockMore contains
1928	   the following header with the data from the each WavpackHeader of the
1929	   WavPack block of the correction file matching the WavPack block in
1930	   the lossy file used to fill the Block data, followed by the data of
1931	   the correction file WavPack block.

1933	   {
1934	     uint32_t crc;           // crc for actual decoded data
1935	     uint32_t blocksize;     // size of the data to follow
1936	   }
1937	   [ correction block data # 1 ]
1938	   {
1939	     uint32_t crc;           // crc for actual decoded data
1940	     uint32_t blocksize;     // size of the data to follow
1941	   }
1942	   [ correction block data # 2 ]
1943	   {
1944	     uint32_t crc;           // crc for actual decoded data
1945	     uint32_t blocksize;     // size of the data to follow
1946	   }
1947	   [ correction block data # 3 ]
1948	   ...

1950	6.  Subtitles

1952	   Here is a list of guidelines for storing subtitles in Matroska:

1954	   *  As a general rule of thumb for all codecs, information that is
1955	      global to an entire stream SHOULD be stored in the CodecPrivate
1956	      element, although not all codec mappings are designed this way.

1958	   *  As subtitles usually come with a start and stop timestamps or a
1959	      start timestamp and a duration, SimpleBlock is usually not used as
1960	      it doesn't allow storing the BlockDuration.  One exception would
1961	      be if the subtitle track has a DefaultDuration which doesn't
1962	      require a BlockDuration.

1964	   *  Start and stop timestamps that are used in a timestamps original
1965	      storage format SHOULD be removed when being placed in Matroska as
1966	      they could interfere if the file is edited afterwards.  Instead,
1967	      the Block's timestamp and BlockDuration SHOULD be used to say when
1968	      the timestamp is displayed.

1970	   *  Because a "subtitle" stream is actually just an overlay stream,
1971	      anything with a transparency layer could be used, including video.

1973	6.1.  Images Subtitles

1975	   A common image format imported into Matroska is the VobSub subtitle
1976	   format.  This subtitle type is generated by exporting the subtitles
1977	   from a DVD [DVD-Video].

1979	   If the subtitle version in the .IDX file is less than v7, the content
1980	   has to be remuxed as the S_VOBSUB CodecID only supports version 7 and
1981	   newer of VobSubs files; see Section 3.5.13.  One way to remux the
1982	   subtitles is to use the SubResync utility from VobSub 2.23 (or MPC)
1983	   into v7 format.  Generally any newly created subs will be in v7
1984	   format.

1986	   The .IFO file will not be used at all.

1988	   If there is more than one subtitle stream in the VobSub set, each
1989	   stream is separated into separate tracks for storage in Matroska.
1990	   E.g. the VobSub file contains streams for both English and German
1991	   subtitles.  Then the resulting Matroska file will contain multiple
1992	   tracks, and language information can be mapped to Matroska's language
1993	   tags and dropped from the streams.

1995	   The .IDX file is reformatted (see below) and placed in the
1996	   CodecPrivate.

1998	   Each .BMP will be stored in its own Block.  The Timestamp will be
1999	   stored in the Block timestamp and the duration will be stored in the
2000	   Default Duration.

2002	   Here is an example .IDX file:

2004	    # VobSub index file, v7 (do not modify this line!)
2005	    #
2006	    # To repair desynchronization, you can insert gaps this way:
2007	    # (it usually happens after vob id changes)
2008	    #
2009	    # delay: [sign]hh:mm:ss:ms
2010	    #
2011	    # Where:
2012	    # [sign]: +, - (optional)
2013	    # hh: hours (0 <= hh)
2014	    # mm/ss: minutes/seconds (0 <= mm/ss <= 59)
2015	    # ms: milliseconds (0 <= ms <= 999)
2016	    #
2017	    # You can also modify timestamps or delete a few subs you don't
2018	    # like. Just make sure they stay in increasing order.

2020	    # Settings

2022	    # Original frame size
2023	    size: 720x480

2025	    # Origin, relative to the upper-left corner, can be overloaded by
2026	    # alignment
2027	    org: 0, 0

2029	    # Image scaling (hor,ver), origin is at the upper-left corner or at
2030	    # the alignment coord (x, y)
2031	    scale: 100%, 100%

2033	    # Alpha blending
2034	    alpha: 100%

2036	    # Smoothing for very blocky images (use OLD for no filtering)
2037	    smooth: OFF

2039	    # In millisecs
2040	    fadein/out: 50, 50

2042	    # Force subtitle placement relative to (org.x, org.y)
2043	    align: OFF at LEFT TOP

2045	    # For correcting non-progressive desync. (in millisecs or
2046	    # hh:mm:ss:ms)
2047	    # Note: Not effective in DirectVobSub, use "delay: ... " instead.
2048	    time offset: 0

2050	    # ON: displays only forced subtitles, OFF: shows everything
2051	    forced subs: OFF

2053	    # The original palette of the DVD
2054	    palette: 000000, 7e7e7e, fbff8b, cb86f1, 7f74b8, e23f06, 0a48ea, \
2055	    b3d65a, 6b92f1, 87f087, c02081, f8d0f4, e3c411, 382201, e8840b, \
2056	    fdfdfd

2058	    # Custom colors (transp idxs and the four colors)
2059	    custom colors: OFF, tridx: 0000, colors: 000000, 000000, 000000, \
2060	    000000

2062	    # Language index in use
2063	    langidx: 0

2065	    # English
2066	    id: en, index: 0
2067	    # Uncomment next line to activate alternative name in DirectVobSub /
2068	    # Windows Media Player 6.x
2069	    # alt: English
2070	    # Vob/Cell ID: 1, 1 (PTS: 0)
2071	    timestamp: 00:00:01:101, filepos: 000000000
2072	    timestamp: 00:00:08:708, filepos: 000001000

2074	   First, lines beginning with "#" are removed.  These are comments to
2075	   make text file editing easier, and as this is not a text file, they
2076	   aren't needed.

2078	   Next remove the "langidx" and "id" lines.  These are used to
2079	   differentiate the subtitle streams and define the language.  As the
2080	   streams will be stored separately anyway, there is no need to
2081	   differentiate them here.  Also, the language setting will be stored
2082	   in the Matroska tags, so there is no need to store it here.

2084	   Finally, the "timestamp" will be used to set the Block's timestamp.
2085	   Once it is set there, there is no need for it to be stored here.
2086	   Also, as it may interfere if the file is edited, it SHOULD NOT be
2087	   stored here and it MUST NOT be used by the decoder.

2089	   Once all of these items are removed, the data to store in the
2090	   CodecPrivate SHOULD look like this:

2092	    size: 720x480
2093	    org: 0, 0
2094	    scale: 100%, 100%
2095	    alpha: 100%
2096	    smooth: OFF
2097	    fadein/out: 50, 50
2098	    align: OFF at LEFT TOP
2099	    time offset: 0
2100	    forced subs: OFF
2101	    palette: 000000, 7e7e7e, fbff8b, cb86f1, 7f74b8, e23f06, 0a48ea, \
2102	    b3d65a, 6b92f1, 87f087, c02081, f8d0f4, e3c411, 382201, e8840b, \
2103	    fdfdfd
2104	    custom colors: OFF, tridx: 0000, colors: 000000, 000000, 000000, \
2105	    000000

2107	   There SHOULD also be two Blocks containing one image each with the
2108	   timestamps "00:00:01:101" and "00:00:08:708".

2110	6.2.  SRT Subtitles

2112	   SRT is perhaps the most basic of all subtitle formats.

2114	   It consists of four parts, all in text:

2116	   1.  A number indicating which subtitle it is in the sequence.

2118	   2.  The time that the subtitle appears on the screen, and then
2119	       disappears.

2121	   3.  The subtitle itself.

2123	   4.  A blank line indicating the start of a new subtitle.

2125	   When placing SRT in Matroska, part 3 is converted to UTF-8 (S_TEXT/
2126	   UTF8) and placed in the data portion of the Block.  Part 2 is used to
2127	   set the timestamp of the Block, and BlockDuration element.  Nothing
2128	   else is used.

2130	   Here is an example SRT file:

2132	   1
2133	   00:02:17,440 --> 00:02:20,375
2134	   Senator, we're making
2135	   our final approach into Coruscant.

2137	   2
2138	   00:02:20,476 --> 00:02:22,501
2139	   Very good, Lieutenant.

2141	   In this example, the text "Senator, we're making our final approach
2142	   into Coruscant." would be converted into UTF-8 and placed in the
2143	   Block.  The timestamp of the block would be set to "00:02:17,440".
2144	   And the BlockDuration element would be set to "00:00:02,935".

2146	   The same is repeated for the next subtitle.

2148	   Because there are no general settings for SRT, the CodecPrivate is
2149	   left blank.

2151	6.3.  SSA/ASS Subtitles

2153	   SSA stands for Sub Station Alpha.  It's the file format used by the
2154	   popular subtitle editor SubStation Alpha.  It allows you to do some
2155	   advanced display features, like positioning, karaoke, or style
2156	   managements...

2158	   For detailed information on SSA/ASS, see the SSA specs [SSA].  It
2159	   includes an SSA specs description and the advanced features added by
2160	   ASS format (standing for Advanced SSA).  Because SSA and ASS are so
2161	   similar, they are treated the same here.

2163	   Like SRT, this format is text based with a particular syntax.

2165	   A file consists of 4 or 5 parts, declared similar to an INI file.

2167	   The first, "[Script Info]" contains some information about the
2168	   subtitle file, such as its title, who created it, type of script and
2169	   "PlayResY", which is very important, because everything in your
2170	   script (font size, positioning) is scaled by it.  Sub Station Alpha
2171	   uses your desktops Y resolution to write this value, so if a friend
2172	   with a large monitor and a high screen resolution gives you an edited
2173	   script, you can mess everything up by saving the script in SSA with
2174	   your low-resolution monitor.

2176	   The second, "[V4 Styles]" or "[V4+ Styles]", is a list of style
2177	   definitions.  A style describes how a text will look on the screen.
2178	   It defines font, font size, primary/.../outline color, position,
2179	   alignment, etc.

2181	   For example, this:

2183	   Format: Name, Fontname, Fontsize, PrimaryColour, SecondaryColour, \
2184	   TertiaryColour, BackColour, Bold, Italic, BorderStyle, Outline, \
2185	   Shadow, Alignment, MarginL, MarginR, MarginV, AlphaLevel, Encoding
2186	   Style: Wolf main,Wolf_Rain,56,15724527,15724527,15724527,4144959,0,\
2187	   0,1,1,2,2,5,5,30,0,0
2188	   The third, "[Events]", is the list of text you want to display at the
2189	   right timing.  You can specify some attributes here, such as the
2190	   style to use for this event (MUST be defined in the list), the
2191	   position of the text (Left, Right, Vertical Margin), or some effect.
2192	   The Name is used by the translator to know who said this sentence.
2193	   Timing is in h:mm:ss.cc (centisec).

2195	   Format: Marked, Start, End, Style, Name, MarginL, MarginR, MarginV, \
2196	   Effect, Text
2197	   Dialogue: Marked=0,0:02:40.65,0:02:41.79,Wolf main,Cher,0000,0000,\
2198	   0000,,Et les enregistrements de ses ondes delta ?
2199	   Dialogue: Marked=0,0:02:42.42,0:02:44.15,Wolf main,autre,0000,0000,\
2200	   0000,,Toujours rien.

2202	   "[Pictures]" or "[Fonts]" part can be found in some SSA files.  These
2203	   parts contain UUE-encoded pictures/font.  These features are only
2204	   used by Sub Station Alpha -- i.e., no filter (Vobsub/Avery Lee
2205	   Subtiler filter) uses them.

2207	   Now, how are they stored in Matroska?

2209	   *  All text is converted to UTF-8

2211	   *  All the headers, "[Script Info]" and the "[V4 Styles]"/"[V4+
2212	      Styles]" list, are stored in CodecPrivate.

2214	   *  Start & End field are used to set TimeStamp and the BlockDuration
2215	      element. the data stored is:

2217	   *  Events are stored in the Block in this order: ReadOrder, Layer,

2304	6.4.  WebVTT

2306	   The "Web Video Text Tracks Format" (short: WebVTT) is developed by
2307	   the World Wide Web Consortium (W3C).  Its specifications are freely
2308	   available at [WebVTT].

2310	   The guiding principles for the storage of WebVTT in Matroska are:

2312	   *  Consistency: store data in a similar way to other subtitle codecs

2314	   *  Simplicity: making decoding and remuxing as easy as possible for
2315	      existing infrastructures

2317	   *  Completeness: keeping as much data as possible from the original
2318	      WebVTT file

2320	6.4.1.  Track Parameters

2322	   The CodecID to use is S_TEXT/WEBVTT.

2324	   This CodecPrivate contains all global blocks before the first
2325	   subtitle entry.  This starts at the "WEBVTT" file identification
2326	   marker but excludes the optional byte order mark.

2328	6.4.2.  Storage of non-global WebVTT blocks

2330	   Non-global WebVTT blocks (e.g., "NOTE") before a WebVTT caption or
2331	   subtitle cue text are stored in Matroska's BlockAddition element
2332	   together with the Matroska Block containing the WebVTT caption or
2333	   subtitle cue text these blocks precede (see below for the actual
2334	   format).

2336	6.4.3.  Storage of Cues in Matroska blocks

2338	   Each WebVTT caption or subtitle cue text is stored directly in the
2339	   Matroska Block.

2341	   A muxer MUST change all WebVTT cue timestamp(s) present within the
2342	   WebVTT caption or subtitle cue text to be relative to the Matroska
2343	   Block's timestamp.

2345	   The Cue's start timestamp is used as the Matroska Block's timestamp.

2347	   The difference between the Cue's end timestamp and its start
2348	   timestamp is used as the Matroska BlockDuration.

2350	6.4.4.  BlockAdditions

2352	   Each Matroska Block may be accompanied by one BlockAdditions element.
2353	   Its format is as follows:

2355	   1.  The first line contains the WebVTT caption or subtitle cue text's
2356	       optional WebVTT cue settings list followed by one line feed
2357	       character (U+0x000a).  The WebVTT cue settings list may be empty,
2358	       in which case the line consists of the line feed character only.

2360	   2.  The second line contains the WebVTT caption or subtitle cue
2361	       text's optional WebVTT cue identifier followed by one line feed
2362	       character (U+0x000a).  The line may be empty indicating that
2363	       there was no WebVTT cue identifier in the source file, in which
2364	       case the line consists of the line feed character only.

2366	   3.  The third and all following lines contain all WebVTT comment
2367	       block(s) that precede the current WebVTT cue block.  These may be
2368	       absent.  Each WebVTT comment block includes its WebVTT line
2369	       terminator and is followed by one line feed character (U+0x000a).
2370	       The last WebVTT comment block MAY omit the WebVTT line terminator
2371	       and the line feed character.

2373	   If there is no Matroska BlockAddition element stored together with
2374	   the Matroska Block, then WebVTT cue settings list, WebVTT cue
2375	   identifier, and WebVTT comment block(s) MUST be assumed to be absent.

2556	6.4.6.  Storage of WebVTT in Matroska vs. WebM

2558	   Note: the storage of WebVTT in Matroska is not the same as the design
2559	   document for storage of WebVTT in WebM [WebM-WebVTT].  There are
2560	   several reasons for this including but not limited to: the WebM
2561	   document is old (from February 2012) and was based on an earlier
2562	   draft of WebVTT and ignores several parts that were added to WebVTT
2563	   later; WebM does still not support subtitles at all [WebMContainer];
2564	   the proposal suggests splitting the information across multiple
2565	   tracks making demuxer's and remuxer's life very difficult.

2567	   WebM uses the "D_WEBVTT/SUBTITLES", "D_WEBVTT/CAPTIONS", "D_WEBVTT/
2568	   DESCRIPTIONS", and "D_WEBVTT/METADATA" CodecID with different tracks
2569	   depending on the data type and without a CodecPrivate.

2571	6.5.  HDMV Presentation Graphics Subtitles

2573	   The specifications for the HDMV Presentation Graphics Subtitle format
2574	   (short: HDMV PGS) can be found in in section 9.14 "HDMV graphics
2575	   streams" of the Blu-ray specifications [Blu-ray.Part3].

2577	6.5.1.  Track Parameters

2579	   The CodecID to use is S_HDMV/PGS.  A CodecPrivate element is not
2580	   used.

2582	6.5.2.  Matroska Blocks

2584	   Each HDMV PGS Segment (short: Segment) will be stored in a Matroska
2585	   Block.  A Segment is the data structure described in section 9.14.2.1
2586	   "Segment coding structure and parameters" of the Blu-ray
2587	   specifications [Blu-ray.Part3].

2589	   Each Segment contains a presentation timestamp.  This timestamp will
2590	   be used as the timestamp for the Matroska Block.

2592	   A Segment is normally shown until a subsequent Segment is
2593	   encountered.  Therefore, the Matroska Block MAY have no Duration.  In
2594	   that case, a player MUST display a Segment within a Matroska Block
2595	   until the next Segment is encountered.

2597	   A muxer MAY use a Duration, e.g., by calculating the distance between
2598	   two subsequent Segments.  If a Matroska Block has a Duration, a
2599	   player MUST display that Segment only for the duration of the
2600	   BlockDuration.

2602	6.6.  HDMV Text Subtitles

2604	   The specifications for the HDMV Text Subtitle format (short: HDMV
2605	   TextST) can be found in section 9.15 "HDMV text subtitle streams" of
2606	   the Blu-ray specifications [Blu-ray.Part3].

2608	6.6.1.  Track Parameters

2610	   The CodecID to use is S_HDMV/TEXTST.

2612	   A CodecPrivate element is required.  It MUST contain the stream's
2613	   Dialog Style Segment as described in section 9.15.4.2 "Dialog Style
2614	   Segment" of the Blu-ray specifications [Blu-ray.Part3].

2616	6.6.2.  Matroska Blocks

2618	   Each HDMV Dialog Presentation Segment (short: Segment) will be stored
2619	   in a Matroska Block.  A Segment is the data structure described in
2620	   section 9.15.4.3 "Dialog presentation segment" of the Blu-ray
2621	   specifications [Blu-ray.Part3].

2623	   Each Segment contains a start and an end presentation timestamp
2624	   (short: start PTS & end PTS).  The start PTS will be used as the
2625	   timestamp for the Matroska Block.  The Matroska Block MUST have a
2626	   Duration, and that Duration is the difference between the end PTS and
2627	   the start PTS.

2629	   A player MUST use the Matroska Block's timestamp and BlockDuration
2630	   instead of the Segment's start and end PTS for determining when and
2631	   how long to show the Segment.

2633	6.6.3.  Character set

2635	   When TextST subtitles are stored inside Matroska, the only allowed
2636	   character set is UTF-8.

2638	   Each HDMV text subtitle stream in a Blu-ray can use one of a handful
2639	   of character sets.  This information is not stored in the MPEG2
2640	   Transport Stream itself but in the accompanying Clip Information
2641	   file.

2643	   Therefore, a muxer MUST parse the accompanying Clip Information file.
2644	   If the information indicates a character set other than UTF-8, it
2645	   MUST re-encode all text Dialog Presentation Segments from the
2646	   indicated character set to UTF-8 prior to storing them in Matroska.

2648	6.7.  Digital Video Broadcasting (DVB) subtitles

2650	   The specifications for the Digital Video Broadcasting subtitle
2651	   bitstream format (short: DVB subtitles) can be found in the
2652	   [ETSI.EN300-743] document.  The storage of DVB subtitles in MPEG
2653	   transport streams is specified in the [ETSI.EN300-468] document.

2655	6.7.1.  Track Parameters

2657	   The CodecID to use is S_DVBSUB.

2659	   The CodecPrivate element is five bytes long and has the following
2660	   structure:

2662	   *  2 bytes: composition page ID (bit string, left bit first)
2663	   *  2 bytes: ancillary page ID (bit string, left bit first)

2665	   *  1 byte: subtitling type (bit string, left bit first)

2667	   The semantics of these bytes are the same as the ones described in
2668	   section 6.2.41 "Subtitling descriptor" of [ETSI.EN300-468].

2670	6.7.2.  Matroska Blocks

2672	   Each Matroska Block consists of one or more DVB Subtitle Segments as
2673	   described in section 7.2 "Syntax and semantics of the subtitling
2674	   segment" of [ETSI.EN300-743].

2676	   Each Matroska Block SHOULD have a Duration indicating how long the
2677	   DVB Subtitle Segments in that Block SHOULD be displayed.

2679	6.8.  ARIB (ISDB) subtitles

2681	   The specifications for the ARIB B-24 subtitle bitstream format
2682	   (short: ARIB subtitles) and its storage in MPEG transport streams can
2683	   be found in the documents [ARIB.STD-B24], [ARIB.STD-B10], and
2684	   [ARIB.TR-B14].

2686	6.8.1.  Track Parameters

2688	   The CodecID to use is S_ARIBSUB.

2690	   The CodecPrivate element is three bytes long and has the following
2691	   structure:

2693	   *  1 byte: component tag (bit string, left bit first)

2695	   *  2 bytes: data component ID (bit string, left bit first)

2697	   The semantics of the component tag are the same as those described in
2698	   [ARIB.STD-B10], part 2, Annex J.  The semantics of the data component
2699	   ID are the same as those described in [ARIB.TR-B14], fascicle 2, Vol.
2700	   3, Section 2, 4.2.8.1.

2702	6.8.2.  Matroska Blocks

2704	   Each Matroska Block consists of a single synchronized PES data
2705	   structure as described in chapter 5 "Independent PES transmission
2706	   protocol" of [ARIB.STD-B24], volume 3, with a
2707	   Synchronized_PES_data_byte block containing one or more ISDB Caption
2708	   Data Groups as described in chapter 9 "Transmission of caption and
2709	   superimpose" of [ARIB.STD-B24], volume 1, part 3.  All of the Caption
2710	   Statement Data Groups in a given Matroska Track MUST use the same
2711	   language index.

2713	   A Data Group is normally shown until a subsequent Group provides
2714	   instructions to clear it.  Therefore, the Matroska Block SHOULD NOT
2715	   have a Duration.  A player SHOULD display a Data Group within a
2716	   Matroska Block until its internal duration elapses, or until a
2717	   subsequent Data Group removes it.

2719	7.  Block Additional Mapping

2721	   Extra data or metadata can be added to each Block using
2722	   BlockAdditional data.  Each BlockAdditional contains a BlockAddID
2723	   that identifies the kind of data it contains.  When the BlockAddID is
2724	   set to "1" the contents of the BlockAdditional element are defined by
2725	   the "Codec BlockAdditions" section of the codec; see Section 3.1.5.

2727	   The following XML depicts the nested elements of a BlockGroup element
2728	   with an example of BlockAdditions with a BlockAddID of "1":

2730	   <BlockGroup>
2731	     <Block>{Binary data of a VP9 video frame in YUV}</Block>
2732	     <BlockAdditions>
2733	       <BlockMore>
2734	         <BlockAddID>1</BlockAddID>
2735	         <BlockAdditional>
2736	           {alpha channel encoding to supplement the VP9 frame}
2737	         </BlockAdditional>
2738	       </BlockMore>
2739	     </BlockAdditions>
2740	   </BlockGroup>

2742	   When the BlockAddID is set a value greater than "1", then the
2743	   contents of the BlockAdditional element are defined by the
2744	   BlockAdditionMapping element, within the associated TrackEntry
2745	   element, where the BlockAddID element of BlockAdditional element
2746	   equals the BlockAddIDValue of the associated TrackEntry's
2747	   BlockAdditionMapping element.  That BlockAdditionMapping element
2748	   identifies a particular Block Additional Mapping by the
2749	   BlockAddIDType.

2751	   The values of BlockAddID that are 2 or greater have no semantic
2752	   meaning, but simply associate the BlockMore element with a
2753	   BlockAdditionMapping of the associated Track.  See Section 7 on Block
2754	   Additional Mappings for more information.

2756	   It is RECOMMENDED to not use a value of 4 for BlockAddID and
2757	   BlockAddIDValue when BlockAddIDType is not 4 -- i.e., ITU T.35
2758	   metadata Section 4.2.3, as some WebM-oriented demuxers consider a
2759	   block with BlockAddID of 4 as ITU T.35 metadata without checking the
2760	   BlockAddIDType element.

2762	   The following XML depicts a use of a Block Additional Mapping to
2763	   associate a timecode value with a Block:

2765	   <Segment>
2766	     <!--Mandatory elements omitted for readability-->
2767	     <Tracks>
2768	       <TrackEntry>
2769	         <TrackNumber>1</TrackNumber>
2770	         <TrackUID>568001708</TrackUID>
2771	         <TrackType>1</TrackType>
2772	         <BlockAdditionMapping>
2773	           <BlockAddIDValue>2</BlockAddIDValue><!--arbitrary value
2774	             used in BlockAddID-->
2775	           <BlockAddIDName>timecode</BlockAddIDName>
2776	           <BlockAddIDType>121</BlockAddIDType>
2777	         </BlockAdditionMapping>
2778	         <CodecID>V_FFV1</CodecID>
2779	         <Video>
2780	           <PixelWidth>1920</PixelWidth>
2781	           <PixelHeight>1080</PixelHeight>
2782	         </Video>
2783	       </TrackEntry>
2784	     </Tracks>
2785	     <Cluster>
2786	       <Timestamp>3000</Timestamp>
2787	       <BlockGroup>
2788	         <Block>{binary video frame}</Block>
2789	         <BlockAdditions>
2790	           <BlockMore>
2791	             <BlockAddID>2</BlockAddID><!--arbitrary value from
2792	               BlockAdditionMapping-->
2793	             <BlockAdditional>01:00:00:00</BlockAdditional><!--presented
2794	              as a string for readability but should use binary encoding
2795	              defined in the associated mapping -->
2796	           </BlockMore>
2797	         </BlockAdditions>
2798	       </BlockGroup>
2799	     </Cluster>
2800	   </Segment>
2801	   Block Additional Mappings detail how additional data is stored in the
2802	   BlockMore element with a BlockAdditionMapping element, within the
2803	   Track element, which identifies the BlockAdditional content.  Block
2804	   Additional Mappings define the BlockAddIDType value reserved to
2805	   identify that type of data as well as providing an optional label
2806	   stored within the BlockAddIDName element.  When the Block Additional
2807	   Mapping is dependent on additional contextual information, then the
2808	   Mapping SHOULD describe how such additional contextual information is
2809	   stored within the BlockAddIDExtraData element.

2811	7.1.  SMPTE ST 12-1 Timecode Description

2813	   SMPTE ST 12-1 timecode values can be stored in the BlockMore element
2814	   to associate the content of a Matroska Block with a particular
2815	   timecode value.  If the Block uses Lacing, the timecode value is
2816	   associated with the first frame of the Lace.

2818	   The Block Additional Mapping contains a full binary representation of
2819	   a 64-bit SMPTE timecode value stored in big-endian format and
2820	   expressed exactly as defined in Section 8 and 9 of SMPTE 12M
2821	   [SMPTE.ST12-1], without the 16-bit synchronization word.  For
2822	   convenience, here are the time address bit assignments as described
2823	   in Section 6.2 of [RFC5484]:

2825	                   +===============+==================+
2826	                   | Bit Positions | Label            |
2827	                   +===============+==================+
2828	                   |      0--3     | Units of frames  |
2829	                   +---------------+------------------+
2830	                   |      8--9     | Tens of frames   |
2831	                   +---------------+------------------+
2832	                   |     16--19    | Units of seconds |
2833	                   +---------------+------------------+
2834	                   |     24--26    | Tens of seconds  |
2835	                   +---------------+------------------+
2836	                   |     32--35    | Units of minutes |
2837	                   +---------------+------------------+
2838	                   |     40--42    | Tens of minutes  |
2839	                   +---------------+------------------+
2840	                   |     48--51    | Units of hours   |
2841	                   +---------------+------------------+
2842	                   |     56--57    | Tens of hours    |
2843	                   +---------------+------------------+

2845	                       Table 3: SMPTE ST 12-1 Time
2846	                          Address Bit Positions

2848	   For example, a timecode value of "07:12:26;18" can be expressed as a
2849	   64-bit SMPTE 12M value as:

2851	   10000000 01100000 01100000 01010000
2852	   00100000 00110000 01110000 00000000

2854	   Or with the irrelevant bits marked with an "x" which gives 26 usable
2855	   bits:

2857	   1000xxxx 01xxxxxx 0110xxxx 010xxxxx
2858	   0010xxxx 001xxxxx 0111xxxx 00xxxxxx

2860	   This is interpreted in hexadecimal:

2862	   *  0x8 units of frames
2863	   *  0x1 tens of frames
2864	   *  0x6 units of seconds
2865	   *  0x2 tens of seconds
2866	   *  0x2 units of minutes
2867	   *  0x1 tens of minutes
2868	   *  0x7 units of hours
2869	   *  0x0 tens of hours

2871	   Given no value is above 9, the BCD coding correspond to the actual
2872	   values:

2874	   *  8 units of frames
2875	   *  1 tens of frames
2876	   *  6 units of seconds
2877	   *  2 tens of seconds
2878	   *  2 units of minutes
2879	   *  1 tens of minutes
2880	   *  7 units of hours
2881	   *  0 tens of hours

2883	   Or:

2885	   *  18 frames
2886	   *  26 seconds
2887	   *  12 minutes
2888	   *  07 hours

2890	8.  Security Considerations

2892	   This document inherits security considerations from the EBML
2893	   [RFC8794] and Matroska [RFC9559] documents.

2895	   Codec handling may be one of the more error-prone aspect of using
2896	   Matroska.  The parsing and interpretation of binary data can lead to
2897	   many types of security issues.  Although these issues don't come from
2898	   Matroska itself, it's worth noting some issues that need to be
2899	   considered.

2901	   The CodecPrivate may be missing from the TrackEntry description.  The
2902	   TrackEntry MAY be discarded in that case.

2904	   An existing CodecPrivate data may be corrupted or incomplete or too
2905	   big.  The TrackEntry MAY be discarded in that case.

2907	   A lot of codec have internal fields to hold values that are already
2908	   found in the TrackEntry like the video dimensions or the audio
2909	   sampling frequency.  If these values differ that can lead to playback
2910	   issues and even crashes.
