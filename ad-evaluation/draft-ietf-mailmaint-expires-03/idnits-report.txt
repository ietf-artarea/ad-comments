8	            Updated Use of the Expires Message Header Field
9	                    draft-ietf-mailmaint-expires-03

11	Abstract

13	   This document allows broader use of the Expires message header field
14	   for mail messages.  Message creators can then indicate when a message
15	   expires, while recipients would use this information to handle an
16	   expired message differently.

65	1.  Introduction

67	   [RFC2156] defined a mapping of header fields between X.400 and
68	   RFC822/MIME.  One of the mapped fields is the "Expires" header field,
69	   which provides a date and time at which a message is considered to
70	   lose its validity.

72	   Netnews articles [RFC5536] have an Expires header with a similar
73	   slightly more strict syntax and similar meaning.

75	   This document extends the use of the "Expires" header field to
76	   Internet email in general, whether the message comes from an X.400
77	   gateway or elsewhere.

79	   The date and time of expiration can be used by the mailbox provider
80	   or the MUA to indicate to the user that certain messages could be de-
81	   emphasized or not shown to the user, to unclutter the user's mailbox.

83	   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
84	   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
85	   "OPTIONAL" in this document are to be interpreted as described in BCP
86	   14 [RFC2119] [RFC8174] when, and only when, they appear in all
87	   capitals, as shown here.

89	   A Message Creator is an agent that generates messages for delivery.
90	   In [RFC5598] parlance, this is an Author or Originator.

92	   A Message Reader is either an agent consuming a message or an agent
93	   storing a message.  In RFC 5589 parlance, this is a Message Store or
94	   a Message User Agent.

96	2.  Defining Expiration

98	   [RFC2156] defined a field called "Expires", which replaced "Expiry-
99	   Date" introduced in [RFC1327].  It did not define the term further,
100	   except to say that no automatic handling past that date can be
101	   expected.  [RFC5536] defined "Expires" for Netnews as a date and time
102	   beyond which the poster deems the article to be no longer relevant
103	   and could usefully be removed, but did not actually require such
104	   removal.  The consensus definition used in this document is that
105	   beyond the stated expiration date, the message "loses its validity".

107	   The header field's use in e-mail has been limited, with no formal
108	   semantic definition to date.  No consensus exists to establish a more
109	   precise definition, in deference to existing implementations.
110	   Accordingly, no additional normative definition is provided here, nor
111	   is any requirement established for any particular handling by Message
112	   Readers.

114	3.  Header Field definition

116	   The header field definition remains the same as in [RFC2156] and
117	   [RFC4021].  It indicates the time at which a message loses its
118	   validity.  Using the ABNF from [RFC5322], its syntax is:

120	   expires = "Expires:" date-time CRLF

122	   Example:

124	   Expires: Wed, 1 Dec 2025 17:22:57 +0000

126	4.  Advice to Message Creators

128	   Message creators add the header field along with a relevant date and
129	   time when they know that the message loses its validity.  This could
130	   apply to commercial newsletters that include time-limited offers,
131	   event announcements, social notifications, and periodic announcement
132	   messages.

134	   Message creators MUST NOT include more than one Expires header field
135	   in a message.

137	5.  Advice to Message Readers

139	   Message readers, such as mailbox providers, web mail and MUAs could
140	   de-emphasize the display of expired messages or not display them.
141	   They could allow users to control the actions to take for expired
142	   messages.

144	   The information provided in the header field is intended to be used
145	   as a signal to provide an improved experience to the end-user.  For
146	   instance, systems might allow automatic rules to clean up expired
147	   email from specific message creators or with defined characteristics,
148	   or to provide a mode to quickly handle all expired email.

150	   Message Transfer Agents (MTAs) MUST NOT discard or reject a message
151	   based solely on the content of this header field, if present.

153	   Automatic deletion of a message bearing an Expires field with a date
154	   and time in the past is NOT RECOMMENDED unless configured by the
155	   mailbox owner.

157	6.  Interoperability Considerations

159	   As "Expires" has never been formally defined for Internet messages
160	   other than those translated from X.400, there might have been
161	   implementations that used this header field name in an incompatible
162	   way.  Though the IETF has never seen such a message, there is a
163	   theoretical risk of confusion.

165	7.  Security considerations

167	   A message creator can put any date in an Expires header field,
168	   including dates in the distant past or future.  Without further
169	   knowledge about the message creator, recipient systems and message
170	   readers cannot assume that the contents of the header are accurate or
171	   benign.

173	   For example, a malicious message creator might send spam mail that
174	   includes a expiry date in the past, in the hope that recipients will
175	   not see or report the mail, and then adaptive spam filters would use
176	   it as non-spam training material.  A creator might include a date in
177	   the immediate future in the hope that a recipient would see and act
178	   on a message, but could not find it later to complain about it.  Or a
179	   creator might include a date in the distant future in the hope that
180	   the message would stay in a recipient's inbox and would be more
181	   likely to be read.

183	   While the header field can be useful to determine how to display a
184	   message to a user, it is unlikely to be useful to determine whether
185	   or not the message is wanted or is fraudulent.
