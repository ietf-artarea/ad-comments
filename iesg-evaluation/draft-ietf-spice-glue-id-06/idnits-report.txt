13	Abstract

15	   This specification establishes a URN namespace for GLobal Unique
16	   Enterprise (GLUE) Identifiers.  This enables URN identifiers to be
17	   used for businesses and organizations.  It enables organizational
18	   identities from existing authorities to be represented within this
19	   URN namespace.

95	1.  Introduction

97	   There are myriad entity identifier types for businesses and
98	   organizations.  With the increasing use of digital credentials, there
99	   is a need for a common methodology for expressing these identifiers
100	   such that claims about and by such entities can be made in a
101	   consistent and interoperable manner.

103	   This specification establishes a URN namespace that standardizes the
104	   expression of existing organizational entity identifiers by providing
105	   a common representation format.  It also establishes an IANA registry
106	   for managing how existing organizational entity identification
107	   mechanisms relate to this namespace.

109	   Any organizational entity identifier whose identification mechanism
110	   has been registered as an Authority Identifier in the registry may be
111	   represented as a GLUE URI.

113	1.1.  Requirements Notation and Conventions

115	   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
116	   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
117	   "OPTIONAL" in this document are to be interpreted as described in
118	   BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all
119	   capitals, as shown here.

121	1.2.  Terminology

123	   This specification uses the following terms:

125	   GLUE URI:  a URI that uses the GLUE URN namespace established in this
126	      specification.

128	   Authority Identifier:  identifier for the External Authority
129	      responsible for assigning the External Identifiers used in GLUE
130	      URIs over which it has jurisdiction.

132	   External Identifier:  identifier assigned by an External Authority to
133	      identify a particular organization within GLUE URIs with Authority
134	      Identifier(s) over which it has jurisdiction.

136	   External Authority:  an organization that allocates External
137	      Identifiers over which it has jurisdiction that are used in GLUE
138	      URIs with its Authority Identifier(s).

140	2.  Core Concepts

142	   Every GLUE URI MUST contain the following components:

144	   *  The Authority Identifier

146	   *  The External Identifier

148	2.1.  Uniqueness and Namespacing

150	   Each GLUE URI MUST be globally unique.

152	   An organization can be identified by multiple GLUE URIs, but each
153	   GLUE URI can only refer to a single organization.

155	   It is assumed that most registered organizational entity
156	   identification schemes already handle any necessary namespacing as
157	   part of the External Identifier.  However, if collisions are possible
158	   within the set of possible external identifiers for an Authority
159	   Identifier scheme, then further namespacing is necessary at the GLUE
160	   URI level.  Such namespacing MUST be done using the Authority
161	   Identifier.  The combination of the Authority Identifier and the
162	   External Identifier MUST result in a unique GLUE URI.

164	   For example, assume there is an External Authority FEA that provides
165	   identifiers for organizations in Singapore and South Korea.  The
166	   identifiers issued in Singapore are unique within Singapore, and the
167	   identifiers issued in South Korea are unique within South Korea, but
168	   there is no guarantee that an organization in Singapore will not be
169	   assigned the same identifier as an organization in South Korea.  Upon
170	   registration of FEA as an Authority Identifier, it would be necessary
171	   to separately register two different Authority Identifiers (e.g.,
172	   FEA-SG and FEA-KR) to provide differentiation between the two sets of
173	   External Identifiers.

175	3.  GLUE URIs

177	   GLUE URIs comply with [RFC3986].  They begin with urn:glue: and are
178	   followed by an Authority Identifier, a colon character (":"), and the
179	   External Identifier allocated by the authority.

181	   Authority Identifiers consist of a sequence of characters beginning
182	   with a letter or digit and followed by any combination of letters,
183	   digits, plus ("+"), hyphen ("-"), or period (".").  Although
184	   Authority Identifiers are case-insensitive, the canonical form is
185	   lowercase and documents that specify Authority Identifiers must do so
186	   with lowercase letters.  An implementation should accept uppercase
187	   letters as equivalent to lowercase in Authority Identifier names
188	   (e.g., allow "EXAMPLE" as well as "example") for the sake of
189	   robustness but should only produce lowercase Authority Identifier
190	   names for consistency.  There is a limit of 50 characters for the
191	   length of an Authority Identifier.  The ABNF [RFC5234] for Authority
192	   Identifiers is:

194	   authority-identifier = (ALPHA/DIGIT) *49( ALPHA / DIGIT / "+" / "-" /
195	   "." )

197	   External Identifiers consist of a sequence of characters beginning
198	   with a letter or digit or hyphen ("-") and followed by any
199	   combination of letters, digits, plus ("+"), hyphen ("-"), or period
200	   (".").  A digit or hyphen is allowed as the first character to permit
201	   the case where the External Identifier is the representation of a
202	   number.  It is specific to the Authority Identifier whether the
203	   External Identifiers are case-insensitive or case-sensitive.  When
204	   they are case-insensitive, the canonical form is lowercase and
205	   documents that specify External Identifiers must do so with lowercase
206	   letters.  There is a limit of 1000 characters for an External
207	   Identifier.  The ABNF [RFC5234] for External Identifiers is:

209	   external-identifier = ( ALPHA / DIGIT / "-" ) *999( ALPHA / DIGIT /
210	   "+" / "-" / "." )

212	   Combining these, the ABNF [RFC5234] for a GLUE URI is:

214	   glue-uri = "urn:glue:" authority-identifier ":" external-identifier

216	   For example, the following is a GLUE URI using the Authority
217	   Identifier "pen" and the External Identifier "32473".  This example
218	   uses the Enterprise Number "32473" reserved for documentation in
219	   [RFC5612].

221	   urn:glue:pen:32473

223	   A GLUE URI is defined over the restricted US-ASCII syntax specified
224	   in this section.  Percent-encoding is not permitted.  Consequently,
225	   GLUE URIs do not support representation of External Identifiers whose
226	   canonical form includes non-ASCII characters.  This specification is
227	   therefore limited to identifier systems whose canonical
228	   representations are fully within the permitted character set.

230	   The Authority Identifier MUST be registered in the GLUE URI Authority
231	   Identifier registry defined in Section 7.1.  The External Identifier
232	   MUST be the identifier assigned to the organization by the External
233	   Authority.

266	4.1.  Equivalence to Similar URIs

268	   A GLUE URI is an identifier in a distinct URN namespace.  By default,
269	   a GLUE URI is not equivalent to any other URI, including a URI
270	   defined by the referenced authority's own namespace.  Equivalence
271	   between a GLUE URI and a non-GLUE URI exists only when explicitly
272	   specified for a given Authority Identifier.  Implementations and
273	   relying parties MUST NOT assume equivalence between GLUE URIs and
274	   non-GLUE URIs unless such equivalence is explicitly defined by the
275	   authority or documented in the relevant registry entry.

277	4.1.1.  LEI URNs

279	   [LEI-IANA] registers a URN namespace for Legal Entity Identifiers
280	   (LEIs).  This means that LEIs can be represented as URNs in at least
281	   two ways.  Therefore there is an equivalence between a GLUE URI with
282	   an "lei" Authority Identifier and an LEI URN, provided the 20-digit
283	   LEI Code of the LEI URN is identifical to the External Identifier of
284	   the GLUE URI.  For example, "urn:lei:INR2EJN1ERAN0W5ZP974" is
285	   equivalent to "urn:glue:lei:INR2EJN1ERAN0W5ZP974".

287	5.  Security Considerations

289	   The security considerations inherent to using URNs apply.  Security
290	   considerations for URNs can be found in [RFC2141].

292	   The global uniqueness of GLUE URIs prevents situations in which the
293	   same identifier is allocated in different local namespaces and cannot
294	   be disambiguated when used.  For instance both Canadian and
295	   Singaporean organization registries might use the local identifier
296	   "42", but with these referring to different organizations.  Embedding
297	   these local identifiers in GLUE URIs enables disambiguation.

299	6.  Privacy Considerations

301	6.1.  Private Identifiers as Corporate Identifiers

303	   There are some corporate identifiers that make use of personal
304	   identifiers.  For example, this is the case for some registered sole-
305	   proprietor businesses in the United States, where the Tax ID may be
306	   the same as the Social Security Number (SSN) of the business owner.
307	   Where the Tax ID uniquely identifies the business, the SSN uniquely
308	   identifies an individual.

310	   It is possible for such business identifiers to be represented as
311	   GLUE URIs.  An identifier's expression as a GLUE URI does not change
312	   the privacy characteristics of that identifier.  The same cautions
313	   and concerns need to be taken with the GLUE URI representation as
314	   with the original identifier.

316	   Implementers storing or evaluating GLUE URIs are encouraged to be
317	   aware the privacy characteristics of each identification scheme
318	   represented by an Authority Identifier and to appropriately handle
319	   any GLUE URI which violates privacy policies.

321	7.  IANA Considerations

323	   This section establishes a registry and populates it with its initial
324	   contents.

326	   Values are registered on a Specification Required (Section 4.6 of
327	   [RFC8126]) basis after a two-week review period on the spice-ext-
328	   review@ietf.org mailing list, on the advice of one or more Designated
329	   Experts.  However, to allow for the allocation of values prior to
330	   publication of the final version of a specification, the Designated
331	   Experts may approve registration once they are satisfied that the
332	   specification will be completed and published.  However, if the
333	   specification is not completed and published in a timely manner, as
334	   determined by the Designated Experts, the Designated Experts may
335	   request that IANA withdraw the registration.

337	   Registration requests sent to the mailing list for review should use
338	   an appropriate subject (e.g., "Request to register URN
339	   urn:glue:example").

341	   Within the review period, the Designated Experts will either approve
342	   or deny the registration request, communicating this decision to the
343	   review list and IANA.  The Designated Experts verify that a
344	   specification exists.  Experts are encouraged to be biased towards
345	   approving registrations unless they are abusive, frivolous, or
346	   actively harmful (not merely aesthetically displeasing or
347	   architecturally dubious).

349	   Denials should include an explanation and, if applicable, suggestions
350	   as to how to make the request successful.  If the designated experts
351	   are not responsive, the registration requesters should contact IANA
352	   to escalate the process.

354	   Criteria that should be applied by the Designated Experts includes
355	   determining whether the proposed registration duplicates existing
356	   functionality, determining whether it is likely to be of general
357	   applicability or whether it is useful only for a single application,
358	   and whether the registration references an existing organizational
359	   registry operated by an External Authority identified by the proposed
360	   Authority Identifier.

362	   IANA must only accept registry updates from the Designated Experts
363	   and should direct all requests for registration to the review mailing
364	   list.

366	   It is suggested that multiple Designated Experts be appointed who are
367	   able to represent the perspectives of different applications using
368	   this specification, in order to enable broadly-informed review of
369	   registration decisions.  In cases where a registration decision could
370	   be perceived as creating a conflict of interest for a particular
371	   Expert, that Expert should defer to the judgment of the other
372	   Experts.

374	   The reason for the use of the mailing list is to enable public review
375	   of registration requests, enabling both Designated Experts and other
376	   interested parties to provide feedback on proposed registrations.
377	   The reason to allow the Designated Experts to allocate values prior
378	   to publication as a final specification is to enable giving authors
379	   of specifications proposing registrations the benefit of review by
380	   the Designated Experts before the specification is completely done,
381	   so that if problems are identified, the authors can iterate and fix
382	   them before publication of the final specification.

384	7.1.  GLUE Authority Identifier URN Registry

386	   This specification establishes the IANA "GLUE Authority Identifier
387	   URN" registry creating a URN namespace for Authority Identifiers for
388	   GLobal Unique Enterprise (GLUE) Identifiers.

390	   Each entry registers the URN for an Authority Identifier within the
391	   "urn:glue:" namespace.  The organization responsible for the
392	   Authority Identifier is recorded.

394	   IANA is requested to create the "GLobal Unique Enterprise (GLUE)
395	   Identifiers" registry group located at
396	   https://www.iana.org/assignments/glue-identifiers/ and place this
397	   registry there.

399	7.1.1.  Registration Template

401	   Authority Identifier:  identifier for the External Authority
402	      responsible for assigning the External Identifier used in GLUE
403	      URIs.  This identifier is not case sensitive and any letters MUST
404	      be expressed in lowercase characters.  It MUST consist of a
405	      sequence of characters with a mazimum length of 50, beginning with
406	      a letter and followed by any combination of letters, digits, plus
407	      ("+"), period ("."), or hyphen ("-").

409	   URN:  The URN within the "urn:glue:" namespace consisting of
410	      "urn:glue:" followed by the Authority Identifier.

412	   Organization:  The organization responsible for the Authority
413	      Identifier.

415	   Change Controller:  For IETF stream RFCs, use "IETF".  For others,
416	      give the name of the responsible party.  Other details (e.g.,
417	      postal address, e-mail address, or home page URI) may also be
418	      included.

420	   Specification Document(s):  Reference to the document or documents
421	      that specify the Authority Identifier to be registered, preferably
422	      including URLs that can be used to retrieve the documents.  An
423	      indication of the relevant sections may also be included, but is
424	      not required.
