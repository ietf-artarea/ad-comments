11	                draft-ietf-netmod-rfc6020-iana-update-01

13	Abstract

15	   This document amends the IANA guidance on the uniqueness of YANG
16	   module and submodule names.

18	   The document updates RFC 6020 to clarify how modules and their
19	   revisions are handled by IANA.

67	1.  Introduction

69	   [RFC6020] defines a registry for YANG module and submodule names,
70	   called "YANG Module Names" [IANA-MOD-NAMES].

72	   Specifically, IANA considerations to register YANG module and
73	   submodule names are specified in Section 14 of [RFC6020].  These
74	   considerations require that all module and submodule names in the
75	   registry must be unique.  However, the practice followed by IANA is
76	   not consistent with that guidance.

78	   This document amends the guidance on the uniqueness of names
79	   (Section 14 of [RFC6020]) to comply with the IANA practices for
80	   registering modules and their revisions.

90	3.  IANA Considerations

92	3.1.  Update YANG Parameters Registry

94	   This document requests IANA to update the reference for the "YANG
95	   Module Names" registry under the "YANG Parameters" registry group
96	   [IANA-MOD-NAMES] to also list to the RFC number that will be assigned
97	   to this document.  This update is needed because the procedure in
98	   this document is authoritative for assigning names in that registry.

100	3.2.  Revisions of Published Modules

102	   This document amends the guidance on the uniqueness of names,
103	   initially defined in Section 14 of [RFC6020], as follows:

105	   OLD:
106	      All module and submodule names in the registry MUST be unique.

108	      All XML namespaces in the registry MUST be unique.

110	   NEW:
111	      Modules and their revisions are maintained in the registry.

113	      All initial version module and submodule names in the registry
114	      MUST be unique.

116	      All XML namespaces of initial version modules in the registry MUST
117	      be unique.

119	      All modules and submodules revisions MUST have the same name as
120	      the one in the initial version of the module and submodule.

122	      All module revisions MUST have the same XML namespace as the
123	      initial version of the module.

125	4.  Operations and Manageability Considerations

127	   This document aligns an IANA policy with the practice for handling
128	   YANG module names (Section 3.2).  As such, there are no new
129	   operations or manageability requirements introduced by this document.

131	5.  Security Considerations

133	   This document defines a new IANA action (Section 3.1) and defines an
134	   update (Section 3.2) to an IANA registration procedure defined in
135	   [RFC6020].  It does not introduce any new or increased security risks
136	   that need to be discussed.
