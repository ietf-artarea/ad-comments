16	Abstract

18	   The YANG model in this document includes three modules, and can be
19	   used to manage network resources and topologies with scheduled
20	   attributes, such as predictable link loss and link connectivity as a
21	   function of time.  The intent is to have this information be utilized
22	   by Time-Variant Routing systems.

82	1.  Overview

84	   YANG [RFC7950] is a data definition language used to define the
85	   contents of a conceptual data store that allows networked devices to
86	   be managed using NETCONF [RFC6241].  YANG is proving relevant beyond
87	   its initial confines, as bindings to other interfaces (e.g., ReST)
88	   and encodings other than XML (e.g., JSON) are being defined.
89	   Furthermore, YANG data models can be used as the basis for
90	   implementation of other interfaces, such as CLI and programmatic
91	   APIs.

93	   There are use cases where changes to the routing topology are
94	   expected as part of network operations.  In some network scenarios,
95	   it's possible to predict the times at which one router will be able
96	   to establish a link with another router.  Links can be predictably
97	   lost and re-established, and neighbors may change as a function of
98	   time.  For examples of such networks and scenarios, please reference
99	   TVR (Time-Variant Routing) Use Cases [RFC9657].

101	   The YANG model in this document can be used to manage network
102	   resources and topologies with scheduled attributes.  There are three
103	   YANG modules in this document.

105	   Module ietf-tvr-schedule.yang contains the schedule YANG definitions.
106	   Module ietf-tvr-topology.yang defines a network topology with time-
107	   variant availability.  Module ietf-tvr-node.yang is to be used to
108	   manage scheduled attributes of a single node.

110	   The YANG modules in this document conform to the Network Management
111	   Datastore Architecture (NMDA) [RFC8342].

126	3.  Design of the model

128	3.1.  Schedule Definitions

130	   Module ietf-tvr-schedule.yang contains schedule definitions that can
131	   be used by other modules.

133	   The grouping "tvr-schedule" consists of a list of schedules, and each
134	   schedule is either a single time period or recurring time periods.
135	   For each schedule instance, it utilizes the groupings "period-of-
136	   time" and "recurrence-utc" as defined in
137	   [I-D.ietf-netmod-schedule-yang].

139	   Users can supply attributes or containers that augment the schedule.
140	   The "tvr-node" has a container "scheduled-attributes" that uses the
141	   "tvr-schedule".

143	   When an attribute's schedule ends, the "value-default" MAY be used,
144	   when present.  Alternatively, values can remain as set until another
145	   schedule alters values.

147	   The following figure provides an illustration of two attributes and
148	   their scheduled value changes.  The attributes A1 and A2 take on
149	   different values at different times.  The attribute A1 will take on
150	   the value v1 from the time t0 until t1, the value v2 from t1 until
151	   t2, and v3 from t2 until t3.  The schedule of A1 may contain a list
152	   of 3 time periods with different values.  The attribute A2 will take
153	   on the value vv1 from time t0 until t1, default value va2 from t1
154	   until t2, and vv2 from t1 until t3.  The schedule of A2 may contain a
155	   list of two time periods, and the default value is used between
156	   scheduled values.

158	   Attributes
159	             +-------------------+----------+------+
160	         A1  |       v1          |   v2     |  v3  |
161	             +-------------------+----------+------+
162	         A2  |      vv1          |  va2     |  vv2 |
163	             +-------------------+-----------------+
164	            t0                  t1        t2      t3
165	                            Time

167	                     Figure 1: Time Varying Properties

169	3.2.  TVR Node YANG Module

171	   Module ietf-tvr-node.yang is a device model and designed to manage a
172	   single node with scheduled attributes.

174	   Each node has scheduled power at the node level.  At the interface
175	   level, a node has a list of interfaces, and each interface has its
176	   own schedule for availability/power up and down, scheduled bandwidth,
177	   and scheduled neighbors.

179	3.3.  TVR Topology YANG Module

181	   Module ietf-tvr-topology.yang describes a network topology with a
182	   time-variant availability schedule.

184	   The module has a list of nodes, identified by a unique "node-id".
185	   Each node has a list of links.  Links are modeled as unidirectional.
186	   Link availability is described from the viewpoint of a particular
187	   source node (the transmitting node) and beginning at a particular
188	   time.  Each link in the list contains the range of times during which
189	   it is available.

191	   The "source-link-id" is a string and used to identify a link as
192	   viewed from the source-node.  Bandwidth and delay are predicted link
193	   attributes.  Delay is the link propagation time and does not include
194	   any queuing delays. "destination-node" of a link may follow a
195	   schedule as well.

836	6.  Operational Considerations

838	   The TVR Schedule YANG modules provide a structured mechanism to
839	   represent time-variant attributes of network elements and topologies.
840	   The following aspects should be considered by operators deploying the
841	   modules.

843	   Operators should be aware that these scheduled attributes represent
844	   planned state changes, not instantaneous configuration changes or
845	   unplanned breakages.

847	   A schedule can be defined using a series of periods or intervals with
848	   recurrence.  The model defined in this document doesn't prevent a
849	   configuration to mix both styles.  However, it's not recommended.

851	   There should be synchronized clocks between controllers and devices.
852	   The time sources (e.g., NTP, PTP) should be accurate to avoid
853	   unintended time/schedule offsets.  Inconsistent time bases can lead
854	   to network misbehavior, such as routing loops or network instability.

856	   The modules are to support TVR use cases where topology and
857	   attributes vary as a function of time.  How control plane and data
858	   plane are impacted by the schedules should be considered by the
859	   operators.

861	   The TVR schedule modules are meant to work together with other models
862	   that manage node or interface attributes.  Operators should consider
863	   the conflicting configurations between the scheduled and non-
864	   scheduled settings and how to handle the conflicts.

866	7.  Security Considerations

868	   The YANG modules specified in this document define a schema for data
869	   that is designed to be accessed via network management protocols such
870	   as NETCONF [RFC6241] or RESTCONF [RFC8040].  The lowest NETCONF layer
871	   is the secure transport layer, and the mandatory-to-implement secure
872	   transport is Secure Shell (SSH) [RFC6242].  The lowest RESTCONF layer
873	   is HTTPS, and the mandatory-to-implement secure transport is TLS
874	   [RFC8446].

876	   The NETCONF access control model [RFC8341] provides the means to
877	   restrict access for particular NETCONF or RESTCONF users to a pre-
878	   configured subset of all available NETCONF or RESTCONF protocol
879	   operations and content.

881	   There are a number of data nodes defined in ietf-tvr-node.yang module
882	   and ietf-tvr-topology.yang that are writable/creatable/deletable
883	   (i.e., config true, which is the default).  These data nodes may be
884	   considered sensitive or vulnerable in some network environments.
885	   Write operations (e.g., edit-config) to these data nodes without
886	   proper protection can have a negative effect on network operations.
887	   There are the subtrees and data nodes and their sensitivity/
888	   vulnerability:

890	      /node-schedule/node-power-schedule
891	      /node-schedule/interface-schedule
892	      /topology-schedule/nodes/available
893	      /topology-schedule/links/available
894	      Modifications to these scheduled attributes may result in a denial
895	      of service.

897	   Some of the readable data nodes in the ietf-tvr-node.yang module and
898	   ietf-tvr-topolgy.yang module may be considered sensitive or
899	   vulnerable in some network environments.  It is thus important to
900	   control read access (e.g., via get, get-config, or notification) to
901	   these data nodes.
