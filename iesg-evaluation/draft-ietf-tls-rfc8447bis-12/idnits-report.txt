12	Abstract

14	   This document updates the changes to TLS and DTLS IANA registries
15	   made in RFC 8447.  It adds a new value "D" for discouraged to the
16	   Recommended column of the selected TLS registries and adds a
17	   "Comments" column to all active registries that do not already have a
18	   "Comments" column.

20	   This document updates the following RFCs: 3749, 5077, 4680, 5246,
21	   5705, 5878, 6520, 7301, and 8447.

93	1.  Introduction

95	   This document instructs IANA to make changes to a number of the IANA
96	   registries related to Transport Layer Security (TLS) and Datagram
97	   Transport Layer Security (DTLS).  These changes update the changes
98	   made in [RFC8447].

100	      |  NOTE for IANA: This document specifies changes to the registry
101	      |  to update the changes made in [RFC8447].

103	   This specification updates the "Recommended" column in TLS registries
104	   to define a third value "D" for items that are discouraged.

106	2.  Terminology

108	   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",

114	3.  Adding "Recommended" Column

116	   The instructions in this document update the Recommended column,
117	   originally added in [RFC8447] to add a third value, "D", indicating
118	   that a value is "Discouraged".  The permitted values of the
119	   "Recommended" column are:

121	   Y:  Indicates that the IETF has consensus that the item is
122	      RECOMMENDED.  This only means that the associated mechanism is fit
123	      for the purpose for which it was defined.  Careful reading of the
124	      documentation for the mechanism is necessary to understand the
125	      applicability of that mechanism.  The IETF could recommend
126	      mechanisms that have limited applicability, but will provide
127	      applicability statements that describe any limitations of the
128	      mechanism or necessary constraints on its use.

130	   N:  Indicates that the item has not been evaluated by the IETF and
131	      that the IETF has made no statement about the suitability of the
132	      associated mechanism.  This does not necessarily mean that the
133	      mechanism is flawed, only that no consensus exists.  The IETF
134	      might have consensus to leave an items marked as "N" on the basis
135	      of its having limited applicability or usage constraints.

137	   D:  Indicates that the item is discouraged.  This marking could be
138	      used to identify mechanisms that might result in problems if they
139	      are used, such as a weak cryptographic algorithm or a mechanism
140	      that might cause interoperability problems in deployment.  When
141	      marking a registry entry as “D”, either the References or the
142	      Comments Column MUST include sufficient information to determine
143	      why the marking has been applied.  Implementers SHOULD consult the
144	      linked references associated with the item to determine the
145	      conditions under which it SHOULD NOT or MUST NOT be used.

147	   Setting a value to "Y" or "D" or transitioning the value from "Y" or
148	   "D" in the "Recommended" column requires IETF Standards Action
149	   [RFC8126] or IESG Approval.  Not all items defined in Standards Track
150	   RFCs need to be set to "Y" or "D".  Any item not otherwise specified
151	   is set to "N".  The column is blank for values that are unassigned or
152	   reserved unless specifically set.

154	3.1.  Recommended Note

156	   Existing registries have a note on the meaning of the Recommended
157	   column.  For the registries discussed in the subsequent sections this
158	   note is updated with a sentence describing the "D" value as follows:

160	   Note:  If "Recommended" column is set to "N", it does not necessarily
161	      mean that it is flawed; rather, it indicates that the item either
162	      has not been through the IETF consensus process, has limited
163	      applicability, or is intended only for specific use cases.  If the
164	      "Recommended" column is set to "D" the item is discouraged and
165	      SHOULD NOT or MUST NOT be used, depending upon the situation;
166	      consult the item’s references for clarity.

211	5.  TLS Cipher Suites Registry

213	   Several categories of ciphersuites are discouraged for general use
214	   and are marked as "D".

216	   Ciphersuites that use NULL encryption do not provide the
217	   confidentiality normally expected of TLS.  Protocols and applications
218	   are often designed to require confidentiality as a security property.
219	   These ciphersuites MUST NOT be used in those cases.

221	   Ciphersuites marked as EXPORT use weak ciphers and were deprecated in
222	   TLS 1.1 [RFC4346].

224	   Cipher suites marked as anon do not provide any authentication and
225	   are vulnerable to man-in-the-middle attacks and are deprecated in TLS
226	   1.1 [RFC4346].

228	   RC4 is a weak cipher and is deprecated in [RFC7465].

230	   DES and IDEA are not considered secure for general use and are
231	   deprecated in [RFC5469].  Nor is MD5 or SHA-1 and these are
232	   deprecated in [RFC9155].

234	   In order to reflect the changes in the Recommended column allocation,

764	15.  Expert Review of Current and Potential IETF and IRTF Documents

766	   The intent of the Specification Required standard for TLS code points
767	   is to allow for easy registration for code points associated with
768	   protocols and algorithms that are not being actively developed inside
769	   IETF or IRTF.  When TLS-based technologies are being developed inside
770	   the IRTF/IETF they should be done in coordination with the TLS WG in
771	   order to provide appropriate review.  For this reason, unless the TLS
772	   WG chairs indicate otherwise via email, designated experts should
773	   decline code point registrations for documents which have already
774	   been adopted or are being proposed for adoption by IETF working
775	   groups or IRTF research groups.

777	16.  Security Considerations

779	   The change to Specification Required from IETF Review lowers the
780	   amount of review provided by the WG for cipher suites and supported
781	   groups.  This change reflects reality in that the WG essentially
782	   provided no cryptographic review of the cipher suites or supported
783	   groups.  This was especially true of national cipher suites.

785	   Recommended algorithms are regarded as secure for general use at the
786	   time of registration; however, cryptographic algorithms and
787	   parameters will be broken or weakened over time.  It is possible that
788	   the "Recommended" status in the registry lags behind the most recent
789	   advances in cryptanalysis.  Implementers and users need to check that
790	   the cryptographic algorithms listed continue to provide the expected
791	   level of security.

793	   Designated experts ensure the specification is publicly available.
794	   They may provide more in-depth reviews.  Their review should not be
795	   taken as an endorsement of the cipher suite, extension, supported
796	   group, etc.
